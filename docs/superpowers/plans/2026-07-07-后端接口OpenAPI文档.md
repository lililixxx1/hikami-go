## 全局技术约定(所有任务遵守,引自 spec §3)

> 这些约定不重复写进每个 task,执行者必须先读本节。

### YAML 风格
- 2 空格缩进,字符串含特殊字符(`:` `#` `{` `}` 等)用双引号包裹
- 每个 path 文件顶层 key 是字面路径(如 `/api/channels`,不是 `api-channels`)
- `$ref` 路径:从 `paths/*.yaml` 引 schema 用 `'../components/schemas/<file>.yaml#/<SchemaName>'`;从 `openapi.yaml` 引 path 用 `'paths/<file>.yaml#/<literal-path>'`
- 路径参数用 `{id}`(OpenAPI 标准),不是 Go 的 `:id`

### 字段类型映射(Go → OpenAPI)
| Go | OpenAPI |
|---|---|
| `string` | `type: string` |
| `int` / `int64` | `type: integer`, `format: int64` |
| `bool` | `type: boolean` |
| `float64` | `type: number`, `format: double` |
| `time.Time` | `type: string`, `format: date-time` |
| `*T` 带 `omitempty` | 字段缺席(不写 `nullable: true`,不进 `required`) |
| `*T` **无** `omitempty`(如 `gin.H{"global": tpl}` 里的 nil 指针) | **`null`**(写 `nullable: true`,字段出现在响应但值为 null)。区分关键:看 json tag 有没有 omitempty。例:`getChannelRecapTemplate` 返回 `{global: Template|null, channel: Template|null}`(server.go:3449,`*Template` 无 omitempty)→ 用 nullable: true;而 `Session.started_at`(`*time.Time` 带 omitempty)→ 字段缺席,不写 nullable。 |

### 通用 schema(每个 path 都可能用)
- `Error`: `{error: string, reason?: string}` — spec §3.3,定义在 common.yaml
- `ErrorWithSessionCount`: 删除频道冲突时用,定义在 common.yaml
- 列表响应约定:后端大量用 `gin.H{"items": [...]}` 包装,OpenAPI 里**每个端点直接定义 inline schema**(如 `{items: [Channel]}`),**不定义泛型 `ListResponse<T>`**(OpenAPI 3.0 不支持泛型)

### 每个 operation 必含
- `tags`(对应 spec §3.2 的 11 个 tag)
- `summary`(简短,中文)
- `operationId`(英文 camelCase,全局唯一,前端生成类型用)
- `responses`(至少一个 2xx + 一个 Error)

- [ ] **Step 2b: 修正主文件 paths 聚合方式**

OpenAPI 3.0 支持 `paths` 下每条路径用 `$ref` 指向外部文件,但被引用文件**顶层 key 必须是该路径字符串**。所以 `paths/channels.yaml` 的顶层是 `/api/channels:` 而非 `api-channels:`。

主文件 `paths` 段最终形态(随每个 task 增量补充):
```yaml
paths:
  /api/healthz:
    $ref: 'paths/system.yaml#/~1api~1healthz'
```

⚠️ **JSON Pointer 转义**:`/` 要转义成 `~1`。这很丑且易错。**采用更简洁的方案**:用 OpenAPI 3.1 的 `$ref` 直接指向外部文件的根 path key,或**放弃跨文件拆分,改用单文件 `paths` + 单独 schema 文件**。

**决策**:鉴于 JSON Pointer 跨文件转义太脆弱,本计划改用**单 paths 段内联聚合**——即 `openapi.yaml` 的 `paths:` 下直接写每个端点的完整定义(不拆到 paths/ 目录),只有 `components/schemas/` 跨文件。这样 `$ref` 只用于引 schema,路径简单。重写文件结构(Step 2c)。

- [ ] **Step 2c: 重写文件结构决策(修正 spec §2.1)**

经评估,spec §2.1 的"paths/ 按域拆分"在 OpenAPI 3.0 下需要难看的 JSON Pointer 转义(`/` → `~1`)。改为:

```
docs/api/
├── openapi.yaml              # 单文件,含 info/paths(全部端点内联)/components 引用
├── components/
│   └── schemas/              # schema 按域拆分(这里 $ref 不需要转义)
│       ├── common.yaml
│       ├── channel.yaml
│       └── ...(15 个)
├── index.html
├── api-gap-analysis.md

```

**权衡**:openapi.yaml 会较大(~2000 行),但可读性远高于跨文件 path 转义。schema 跨文件是安全的(`$ref: 'components/schemas/channel.yaml#/Channel'`,无特殊字符)。

**这是对 spec §2.1 的执行期修正**,记入 README。
