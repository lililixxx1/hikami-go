# Windows 桌面模式：子进程闪窗问题处理报告

> **文档版本**：2026-07-18（事后报告 post-mortem 风格；状态字段于落地后回填）
> **状态**：✅ **已修复并验证落地**（branch `fix/investigations-2026-07-18`，见 §6）
> **适用范围**：`make build-windows-desktop` / `make build-windows-desktop-lite` 产出的 exe
> **用户反馈**："软件在后台运行时会有终端窗口突然闪现又关闭"

---

## 1. 问题现象

### 1.1 触发条件

仅在 **Windows 桌面模式**（用 `-ldflags='-H windowsgui'` 编译、`-tags` 含 `systray`）下运行 exe 时复现。

对应 Makefile target：
- `build-windows-desktop`（`Makefile:63-65`，`-tags 'embed_ffmpeg,embedded_web,systray'`）
- `build-windows-desktop-lite`（`Makefile:69-71`，`-tags 'embedded_web,systray'`）

非桌面版的 `build-windows-amd64` / `-lite` **不复现**（这两个 target 没有 `-H windowsgui`，主进程本身就是控制台程序，子进程复用同一控制台，没有"新开窗口"动作）。

### 1.2 现象描述

软件驻留后台（托盘图标存在）期间，**黑色控制台窗口间歇性闪现、瞬间关闭**。用户观察到：

- 无操作时也可能突然闪（后台自动任务运行中）；
- 每次持续时间 < 1 秒（短子进程）至 数秒（长子进程启动瞬间）；
- 与托盘"打开管理界面"操作关联时（点菜单后闪一次 cmd 窗）。

### 1.3 现象与业务流水线的对应

按业务流水线（[docs/data-flow.md](./data-flow.md)）复盘，回顾生成一场直播会按顺序派生多个子进程。**每场结束的回顾流水线会让窗口连续闪 5~8 次**：

| 流水线阶段 | 子进程 | 闪窗次数 |
|----------|--------|---------|
| download（多P 分段下载，仅回放类） | `yt-dlp` × N | N |
| download（多P 合并） | `ffmpeg -c copy` | 1 |
| download（时长探测） | `ffprobe` | 1 |
| normalize（转 16kHz mp3） | `ffmpeg` | 1 |
| asr（DashScope，临时文件走 rclone） | `rclone copyto` + `rclone deletefile` | 2 |
| recap（外部 CLI 模式，可选） | `claude` 或 `codex` | 1 |
| upload（上传网盘） | `rclone copy` + `rclone delete` | 2 |

直播录制期间 `ffmpeg` 作为长子进程会"一直挂着"一个 ffmpeg 窗口（不是"闪"，是"开在那里"，但启动瞬间也会闪一下）——这是最扰人的表现。

## 2. 根因分析

### 2.1 直接原因：子进程未设置 `CREATE_NO_WINDOW`

对整个仓库（排除 `web/dist`）做关键字扫描：

```bash
grep -rn 'SysProcAttr|HideWindow|CREATE_NO_WINDOW|CreationFlags' --include='*.go'
```

**零匹配**（修复前）。即所有 `exec.Command` / `exec.CommandContext` 调用都未配置 Windows 进程创建标志。

### 2.2 机制：GUI 子系统 + 控制台子进程 = 新建控制台

Windows 进程创建标志的语义（[`CreateProcess` dwCreationFlags](https://learn.microsoft.com/en-us/windows/win32/procthreadprocess-creation-flags)）：

| 父进程控制台状态 | 子进程类型 | 无 CreationFlags 时 Windows 行为 |
|----------------|-----------|--------------------------------|
| 有控制台（普通 console exe） | 控制台程序 | **复用父控制台**，不新建窗口 |
| **无控制台**（`-H windowsgui`） | 控制台程序 | **为子进程新建一个控制台窗口** ← 本问题 |
| 无控制台 | 控制台程序 + `CREATE_NO_WINDOW` (0x08000000) | 无任何窗口，pipe 正常工作 ← **修复目标** |

桌面模式用 `-H windowsgui` 把主程序链为 **Windows GUI 子系统**（PE 头 `IMAGE_SUBSYSTEM_WINDOWS_GUI`），系统不为它分配控制台——这正是"双击 exe 无黑窗"的实现机制。但代价是：**所有派生的控制台子进程都会触发 Windows 的"父子控制台不匹配 → 新建控制台"行为**，子进程退出时窗口随之关闭，表现为闪窗。

### 2.3 调用点全景（修复前）

仓库无共享 exec helper（`grep execHelper|runCommand|hideWindow` 零匹配），调用点分散在 9 个包，共 **20 处生产调用**（+ 2 处测试）：

即：**这是 2026-07-14 桌面模式引入的回归问题**，潜伏约 4 天，未在 changelog 中提及。

## 3. 影响评估

### 3.1 严重度：中

| 维度 | 评估 |
|-----|------|
| 功能正确性 | **无影响**。子进程 stdin/stdout/stderr pipe 不受闪窗影响，业务流程完全正常。 |
| 用户体验 | **明显干扰**。每场回顾流水线连续闪 5~8 次，直播录制期间常驻 ffmpeg 窗口。 |
| 数据安全 | 无影响。 |
| 性能 | 无影响（创建/销毁控制台开销 < 10ms）。 |
| 安全 | 无直接影响。理论上控制台窗口会暴露子进程 stdout 给同 session 用户，但桌面单用户场景无实际泄露面。 |

### 3.2 受影响用户

- **桌面模式用户**（Windows 单文件 exe 双击运行，依赖托盘）：全部受影响。
- **服务器/systemd 部署**（Linux）：**不受影响**（非 Windows，且 `-H windowsgui` 无效）。
- **Windows 命令行模式用户**（`build-windows-amd64`，在 cmd/PowerShell 里跑）：不受影响（主进程有控制台）。

### 3.3 回归风险（修复时）

修复需要在 20 处调用点加 helper 调用。风险评估：

- **`CREATE_NO_WINDOW` 与 pipe 兼容**：该 flag 仅抑制控制台窗口创建，**不影响** stdin/stdout/stderr 重定向。所有调用点现有 pipe 行为（`CombinedOutput()` / `StdinPipe()` / `Stdout=&buf` 等）保持不变。
- **`CREATE_NO_WINDOW` 与 `cmd.Cancel = SIGTERM`**：`cmd.Cancel` 是 Go 运行时层面的逻辑（通过 `cmd.Process.Signal`），与 Windows 进程创建标志正交，互不影响。`live_record/ffmpeg.go` 的优雅停止行为保留。
- **跨平台**：`CREATE_NO_WINDOW` 常量在 `syscall` 包未导出（详见 §5.2），必须用 build constraint 拆分（`exec_windows.go` / `exec_other.go`），不能直接在跨平台文件里引用。
- **测试覆盖**：现有测试用桩替换了多数子进程调用（如 `RecordWithProcessStart`、`runFFmpegConcat` 是 var），helper 改动不影响测试桩逻辑；新增 helper 本身在非 Windows 平台是 no-op，无需新增测试。

## 5. 修复方案

### 5.1 方案选型

| 方案 | 描述 | 优点 | 缺点 | 采用 |
|-----|------|-----|------|-----|
| **A. 共享 helper + 收敛调用点** | `executil.HideWindow(cmd)`，调用点加一行 | 改动机械、可读性好、未来新调用点易遵循 | 多文件改动 | ✅ |
| B. 直接 inline `SysProcAttr` | 每处直接写 `cmd.SysProcAttr = &syscall.SysProcAttr{...}` | 无抽象 | 代码重复、build constraint 要拆 20 次、不可维护 | ✗ |
| C. 包装 `exec.Command` 为 `executil.Command` | 替换所有 `exec.Command` 调用 | 单点收敛 | 改动面最大、违反"标准库优先"、`exec.CommandContext` 也要包装 | ✗ |
| D. `DETACHED_PROCESS` (0x00000008) | 子进程脱离父进程控制台 | 也能隐窗 | **会破坏 stdin/stdout pipe**（与 `CombinedOutput`/`StdinPipe` 冲突） | ✗ |

**采用方案 A**。

### 5.2 关键实现细节

#### 5.2.1 helper 位置：`internal/executil/`（非最初推荐的 `runtime`）

**最初规划**：分析阶段推荐把 helper 放在 `internal/runtime/`（该包已承担"能力探测 + 命令解析"职责，函数名 `HideWindow` 聚焦）。

**实施时发现 import cycle**：
```
runtime/probe.go  →  导入 internal/asr  （能力探测）
asr/dashscope.go  →  反向导入 internal/runtime  （若 helper 放此）
                                 ❌ import cycle not allowed
```

**修正方案**：新建**零依赖**的底层小包 `internal/executil/`，符合本项目"聚焦的小包"惯例（参照 `fsutil`、`biliutil`）。包名贴近 Go 标准库 `os/exec` 语义，未来可承载更多 exec 辅助。

#### 5.2.2 `CREATE_NO_WINDOW` 常量未导出

Go 标准库 `syscall` 包**未导出** `CREATE_NO_WINDOW`（仅含部分 CreationFlags）。在 `exec_windows.go` 内按 Win32 头文件值声明：

```go
const CREATE_NO_WINDOW = 0x08000000
```

参考：https://learn.microsoft.com/en-us/windows/win32/procthread/process-creation-flags

#### 5.2.3 改造模式

每处统一在 `cmd := exec.Command(...)` 之后、`Start/Run/Output/CombinedOutput` 之前加一行：

```go
cmd := exec.CommandContext(ctx, command, args...)
executil.HideWindow(cmd)   // ← 新增
// ...（原有 Stdin/Stdout/Cancel 等配置保留）
if err := cmd.Start(); err != nil { ... }
```

`live_record/ffmpeg.go:38` 的特殊处理：`cmd.Cancel = ... SIGTERM` 这行**保持不动**，`HideWindow` 在它之前调用即可。两者正交：`cmd.Cancel` 走 Go 运行时的 `cmd.Process.Signal`，与 Windows 进程创建标志独立。

### 5.3 新增 helper 文件

**`internal/executil/exec_windows.go`**：

```go
//go:build windows

package executil

import (
	"os/exec"
	"syscall"
)

// CREATE_NO_WINDOW 是 Windows CreateProcess 的 dwCreationFlags 位，
// 抑制为子进程分配新控制台（从而避免黑色窗口闪现）。
// 标准库 syscall 未导出该常量（仅含部分 CreationFlags），此处按 Win32 头文件值声明。
// 参考：https://learn.microsoft.com/en-us/windows/win32/procthread/process-creation-flags
const CREATE_NO_WINDOW = 0x08000000

// HideWindow 配置 cmd 在运行时不弹出可见的控制台窗口。
//
// 仅 Windows 有效。当父进程用 -H windowsgui（GUI 子系统、无控制台）构建时必需：
// 在该模式下派生控制台子进程会让 Windows 为子进程新建一个控制台，表现为黑色窗口闪现。
// CREATE_NO_WINDOW 抑制该分配，同时保持 stdin/stdout/stderr pipe 行为不变。
//
// cmd.SysProcAttr 已设置时也安全调用：仅 OR 进该 flag。
// 非 Windows 平台为 no-op（见 exec_other.go）。
//
// 与 cmd.Cancel（如 ffmpeg 录制的 SIGTERM 优雅停止）正交，互不影响：
// cmd.Cancel 走 Go 运行时的 cmd.Process.Signal，与 Windows 进程创建标志独立。
func HideWindow(cmd *exec.Cmd) {
	if cmd.SysProcAttr == nil {
		cmd.SysProcAttr = &syscall.SysProcAttr{}
	}
	cmd.SysProcAttr.CreationFlags |= CREATE_NO_WINDOW
}
```

**`internal/executil/exec_other.go`**：

```go
//go:build !windows

package executil

import "os/exec"

// HideWindow 在非 Windows 平台为 no-op。背景与原理见 exec_windows.go。
func HideWindow(cmd *exec.Cmd) {}
```

## 7. 设计要点与最佳实践

1. **`HideWindow(cmd)` 安全性**：仅 OR 进 `CREATE_NO_WINDOW` 位，不覆盖已有 `SysProcAttr`；与 `cmd.Cancel`（如 ffmpeg 录制的 SIGTERM 优雅停止）正交，互不影响。
2. **与 pipe 兼容**：`CREATE_NO_WINDOW` 仅抑制控制台分配，不影响 stdin/stdout/stderr 重定向，所有调用点现有 `CombinedOutput()` / `StdinPipe()` / `Stdout=&buf` 行为保持不变。
3. **跨平台 no-op**：非 Windows 平台 `HideWindow` 为空函数，零运行时开销。
4. **`live_record/ffmpeg.go` 录制长子进程**：`HideWindow(cmd)` 在 `cmd.Cancel = ... SIGTERM` **之前**调用，保持 Cancel 回调不变。修复前会看到一个常驻 ffmpeg 窗口（一场直播几小时），修复后完全隐藏。
5. **未来新调用点约定**：任何新增 `exec.Command` / `exec.CommandContext` 派生外部进程的代码，都应在 `Start/Run/Output/CombinedOutput` 之前调用 `executil.HideWindow(cmd)`，确保桌面模式下不闪窗。

## 8. 备查：与"2026-07-13 exe 闪退"的区分

项目历史上有一份 [`docs/exe闪退问题与修复.md`](./exe闪退问题与修复.md)，记录的是 2026-07-13 的"双击 exe 一闪而过"问题。两者表现相似但**根因完全不同**，避免混淆：

| 维度 | 2026-07-13 exe 闪退 | 本次（2026-07-18）子进程闪窗 |
|-----|--------------------|-----------------------------|
| 现象 | **exe 本身启动后立即退出**，窗口关太快看似"闪退" | exe 正常驻留，运行中**子进程**窗口闪现 |
| 是否影响功能 | 是（程序根本起不来） | 否（功能正常） |
| 触发时机 | 双击启动瞬间 | 业务运行中（录制/回顾/上传） |
| 根因 | 裁剪版 ffmpeg zip 结构与 `ffmpeg_manifest.go` 路径不匹配，`finalizeInstall` 找不到二进制 → 启动 health check fatal → `exit(1)` | 桌面模式 `-H windowsgui` 下，派生控制台子进程未设置 `CREATE_NO_WINDOW`，Windows 为子进程新建控制台 |
| 涉及文件 | `internal/runtime/ffmpeg_manifest.go` | 18 处 `exec.Command` 调用点 + 新增 `internal/executil/` |
| 修复 | manifest 路径对齐裁剪版 zip 结构 | `executil.HideWindow(cmd)` + build constraint 拆分 |
