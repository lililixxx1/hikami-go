## 五、本轮修复确认（验证上两轮发现的缺陷均已修复）

### 缺陷 A（第 1 轮 file 协议 / 第 2 轮 aresample filter）

```
$ ffmpeg -hide_banner -filters | grep -w aresample
 ... aresample         A->A       Resample audio data.        ← 已启用
```

用例 2 输出 9359 bytes 真实 mp3 产物，证明 libmp3lame + aresample 链路通畅。

### 缺陷 B（第 2 轮 .m4a 扩展名推断失败）

```
$ ffmpeg -hide_banner -muxers | grep -w ipod
  E  ipod            iPod H.264 MP4 (MPEG-4 Part 14)         ← 已启用

$ ffmpeg -hide_banner -h muxer=ipod | head -4
Muxer ipod [iPod H.264 MP4 (MPEG-4 Part 14)]:
    Common extensions: m4v,m4a,m4b.                           ← .m4a 被认领
```

ipod muxer 注册时声明扩展名含 `m4a`，现在 `.m4a` 输出能自动推断到 mov 家族。
用例 3/4/5 输出真实的 .m4a 产物（10132 / 9490 / 5136 bytes），全部通过。

## 六、异常说明

1. **用例 6 的"跳过"为设计如此，非问题。**
   裁剪版的 flv muxer（写）未启用，因为软件只需**读取** B 站直播的 FLV 流，
   不需要写 FLV。脚本因此无法用裁剪版自己合成 FLV 测试素材，遂跳过此用例。
   flv demuxer（读）已在 configure 白名单，真实直播录制可正常解析输入。
   按交接文档"异常 3"约定：**PASS≥6 且 FAIL=0 即算通过**，本项跳过不影响判定。

2. **`verify-ffmpeg-minimal.sh` 本身也已按前两轮反馈修复：**
   - 失败用例现在会 dump 真实 stderr（不再 `2>/dev/null` 吞掉）
   - concat list 改用相对路径（修了 Git Bash 下 `$(pwd)` 产生 MSYS `/c/...` 路径、
     Windows ffmpeg 不认的 bug）
   本轮全绿，这两处脚本修复未被触发，但保留了，后续重编译回归时会更易定位问题。

3. **`Windows验证任务交接.md` 第三节"情况 1"措辞仍建议修正（同前两轮）。**
   文档说形式 A 直接 `bash verify-ffmpeg-minimal.sh` 即可"自动解压 ffmpeg.zip"，
   但脚本实际找的 zip 路径是 `${REPO_ROOT}/internal/runtime/assets/ffmpeg.zip`
   （脚本上级目录的 `internal/runtime/assets/`），不是与脚本同目录的 ffmpeg.zip。
   本轮实际改用 `FFMPEG_BIN=... bash verify-ffmpeg-minimal.sh`，不影响结论。
