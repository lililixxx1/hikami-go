# 待办计划:剩余 B 站 API 端点 -352 风控加固

> **状态**:📋 待办(尚未实施)
> **创建**:2026-07-05
> **关联已落地**:[identify -352 修复 + buvid 共享下沉](../../../internal/biliutil/buvid.go)（已实施 + codex 审核 APPROVED）
> **目标**:把 2026-07-05 identify 修复时发现、但按 systematic-debugging 纪律未触碰的**其余 -352 风险端点**，统一接入已有的共享风控对抗层（`biliutil.BuvidStore` + `biliutil.WBISigner` + `BiliUserAgent`），消除"会随 B 站风控收紧而静默失败"的隐患。
>
> **为什么是待办而非立即修**:这些端点**目前未报告问题**（用户无反馈、无错误日志），盲目修改已工作代码违反 systematic-debugging 的"不猜测、不扩大范围"纪律。本文档为后续单开任务做准备，含完整证据与最小改动方案。

## 一、背景:为什么会有"剩余端点"

2026-07-05 修复 identify 主播识别的 -352 时，做了三件事：
1. 新增 `internal/biliutil/buvid.go`：`BuvidStore`（按 cookie 24h 缓存 buvid3/buvid4，nil-safe）+ `InjectBuvids`（replace 语义注入）。
2. identify 用 buvid 注入 + WBI 签名 + 浏览器 UA/Referer/Origin 通过风控。
3. publisher / live_record.danmaku 的本地 buvid 实现下沉到共享 `BuvidStore`（行为等价重构）。

排查过程中用 Explore agent 全仓扫描了所有"直接调用 `*.bilibili.com` 域名 API"的代码点，发现 **除已修的 identify、已对抗的 publisher/danmaku 外，还有 4 处端点缺风控对抗**。它们当前能用，只是因为：① B 站风控强度按端点/账号状态不同；② 部分端点恰好靠 UA+Referer+Cookie 暂时够用。一旦风控收紧（如某账号触发风险、B 站策略调整），这些点会像 identify 当初一样静默返回 -352，表现为功能失效且难排查。

## 二、待加固端点清单（按风险排序）

### 🔴 高风险 1:`internal/live_record/bilibili.go` — getInfoByRoom / getRoomPlayInfo

## 四、实施建议

### 优先级
1. **P0（强烈建议先做）**:`live_record/bilibili.go` 的 `getInfoByRoom`——与 identify 同端点，且后果是**漏录直播**（用户最先感知、最难恢复）。改动小、方案已验证。
2. **P1**:`biliutil/video.go` 的 `view`——风控最严端点，影响回放下载。
3. **P2**:`handler/server.go` 两个内联裸调——非核心路径，可合并改造。

### 实施纪律(systematic-debugging)
- **每个端点单独开任务**，不打包一次性改 4 个（隔离回归）。
- **先复现确认风险**：上线前用目标账号 + curl 裸请求该端点，确认当前是否真能过。若已能稳定过（说明风控对该端点宽松），可降优先级或只补防御性头。
- **行为等价优先**：buvid/WBI 注入失败时降级为不带（仅 warn），与 identify/publisher 容错策略一致，避免风控对抗组件本身成为单点故障。
- **复用共享层**：所有修复走 `biliutil.BuvidStore` + `biliutil.WBISigner`，禁止再造本地实现（2026-07-05 已消除 publisher/danmaku 两份重复，不要倒退）。
- **codex 审核**：每端点改完走 codex review（`codex review --uncommitted` 或 `codex exec` 审 diff）。

### 通用修复模板（按此套用到每个端点）
```go
// 1. 注入 buvid（共享 BuvidStore，nil-safe，失败降级）
if b3, b4, err := c.buvids.GetBuvids(ctx, cookieHeader); err != nil {
    slog.Warn("xxx: get buvids failed, continuing without buvid", "error", err)
} else if b3 != "" || b4 != "" {
    cookieHeader = biliutil.InjectBuvids(cookieHeader, b3, b4)
}
// 2. WBI 签名（按 cookie 缓存 signer，失败降级为不签名）
if signed, err := c.signerForCookie(cookieHeader).SignURL(endpoint); err == nil {
    endpoint = signed
} else {
    slog.Warn("xxx: wbi sign failed, continuing unsigned", "error", err)
}
// 3. 请求头（真实浏览器 UA + Referer/Origin）
req.Header.Set("User-Agent", biliutil.BiliUserAgent)
req.Header.Set("Referer", "https://live.bilibili.com")   // 或对应子域
req.Header.Set("Origin", "https://live.bilibili.com")
```
