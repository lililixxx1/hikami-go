## 一、任务分工

### 你(用户)做的
1. 去 B 站直播页面找 **5 个正在直播的主播**(任意分区,电台/音乐类最稳)。

4. 加完后告诉 agent,开始监控。

### Agent 做的
- 写后台监控脚本(见第三节),每 60 秒查一次。
- 异常时告警并报告。
- 发现 bug 时按 **收集证据→停机→根因分析(systematic-debugging)→修复→重跑** 处理。

## 二、为什么不是"10 个主播 6 小时"

原计划"10 个主播 6 小时"有几个问题(已与用户讨论确认):
- ❌ 难找 10 个同时在线且连续播 6 小时的主播。
- ❌ 录 10 路对 B 站并发拉流可能触发限流/-352,污染测试。
- ⚠️ 自然流断频率不可控,可能跑很久碰不到重连类 bug。
- ⚠️ "每 10 分钟人工轮询"低效(6h = 36 次),改用后台脚本自动监控、异常才告警。

**最终方案**:5 个真实主播 + 后台监控脚本(60s 轮询)。先跑起来观察,不预设时长——稳定判据见第五节。

## 三、监控脚本

### 脚本位置
`/tmp/hikami-monitor.sh`(临时文件,不进仓库)

### 脚本功能(每 60 秒)
1. **日志告警信号**:grep `/tmp/hikami.log` 里的关键 ERROR/WARN:
   - `stream ended cleanly while still live` / `stream interrupted` → 流断/重连触发
   - `reconnect stream selection failed` / `reconnect liveness probe failed` → 重连失败
   - `recording unhealthy: file not growing` → ffmpeg 卡死/漏录(连续 3 轮 ~3min 文件不增长)
   - `danmaku record failed` / `danmaku websocket host failed` → 弹幕彻底断
   - `scheduler: (discovery|live check) failed` → 整轮调度失败(全体不录)
   - `risk control (-352)` → 风控触发
2. **DB 状态**:`sqlite3 hikami.db` 查:
   - 当前 `status='recording'` 的场次数 + 明细(channel/title/已录分钟)
   - 最近失败场次(`status='failed'`,含 `last_error`)
   - 录制时长异常短(<5min)的场次(疑似漏录)
   - 同频道分段数 >1(疑似流断切成多段)
3. **ffmpeg 进程数**:`pgrep -af ffmpeg | grep -cE 'pipe:0|-c:a copy'`,对比 DB 录制中数(应相等;ffmpeg >> DB 录制数 = 进程泄漏/孤儿)。
4. **输出**:每次扫描的摘要写到 `/tmp/hikami-monitor.log`;发现异常时高亮 `⚠️ ALERT` 行。

### 关键日志信号对照表(来自 Explore 调研)

| msg(节选) | 含义 | 严重度 |
|---|---|---|
| `stream ended cleanly while still live, attempting reconnect` | ffmpeg 干净 EOF 但主播仍在线 → 触发重连(典型流断) | ⚠️ 关注 |
| `stream interrupted, attempting reconnect` | ffmpeg 报错且主播仍在线 → 重连 | ⚠️ 关注 |
| `reconnect stream selection failed` | 重连时 selectStream 失败,本次重连作废 | 🔴 重连失败 |
| `recording unhealthy: file not growing` | 连续 ~3min 输出文件未增长(ffmpeg 卡死/漏录) | 🔴 漏录 |
| `preflight live check failed, proceed optimistically` | 录制前 CheckLive 探测失败(-352/网络),仍乐观尝试 | ⚠️ 漏录前兆 |
| `danmaku websocket host failed` | 弹幕所有地址 WS 连接失败 | 🟡 弹幕断(音频可能正常) |
| `scheduler: live check failed` | 整轮直播检查失败(全体本轮不录) | 🔴 严重 |
| `disk usage high` | 磁盘 ≥85%(将导致写盘失败) | 🔴 严重 |

### DB 监控 SQL(可直接用)

```sql
-- 当前录制中
SELECT COUNT(*) FROM sessions WHERE source_type='live_record' AND status='recording';

-- 当前录制明细(含已录分钟)
SELECT channel_id, title, started_at,
       ROUND((strftime('%s','now')-strftime('%s',started_at))/60.0,1) AS minutes
FROM sessions WHERE source_type='live_record' AND status='recording';

-- 最近失败场次
SELECT channel_id, title, started_at, ended_at, last_error
FROM sessions WHERE source_type='live_record' AND status='failed'
  AND updated_at >= datetime('now','-1 hour')
ORDER BY updated_at DESC;

-- 录制时长异常短(<5min = 疑似漏录)
SELECT channel_id, title, started_at, ended_at,
       ROUND((strftime('%s',ended_at)-strftime('%s',started_at))/60.0,1) AS minutes,
       last_error
FROM sessions WHERE source_type='live_record'
  AND started_at IS NOT NULL AND ended_at IS NOT NULL
  AND (strftime('%s',ended_at)-strftime('%s',started_at)) < 300
ORDER BY started_at DESC LIMIT 20;
```

## 四、"稳定"判据(退出条件)

连续观察至少 **2 小时**,满足以下**全部**条件即认为当前稳定:
1. 0 条 🔴 严重信号(`recording unhealthy` / `reconnect failed` / `scheduler failed` / `disk high`)。
2. `stream ended cleanly` / `stream interrupted` 出现时,后续有对应的 `live record start requested`(重连成功恢复)。
3. ffmpeg 进程数 ≈ DB `status='recording'` 数(无泄漏/孤儿)。
4. 录制时长异常短(<5min)的场次数 = 0(或可解释为主播自然下播)。

> 注:真实主播会自然下播/上播,这不是 bug。判据关注的是"录制过程中的异常",不是"主播在不在线"。

## 五、发现 bug 时的处理流程(systematic-debugging)

```
发现异常 → 抓现场(日志/DB/ps 快照) → 停当前测试
  → 根因分析(Phase 1-2) → 修复(Phase 3-4,TDD)
  → make test 全过 → 重编重启 → 重新开始观察
```

- 不"边跑边热修"(避免重启打断测试 + 补丁式修复引入新 bug)。
- 每次修复走 codex 审核实际代码(`codex exec` 审 diff)。
- 修复后同步文档(模块 CLAUDE.md + 根 CLAUDE.md/AGENTS.md changelog)。
