# 录播稳定性测试 — 完整总结

> **测试周期**:2026-07-04 ~ 2026-07-06(3 天)
> **测试背景**:添加 17 个录播频道(只录播,关闭 ASR/回顾/发布)后,系统性验证录播稳定性,参考 bililive-go / biliLive-tools 两个开源项目。
> **测试方法**:黑盒观察 + 日志/DB 取证 + 根因定位 + TDD 修复 + codex 审核 + 部署验证。
> **结果**:发现 **11 个异常**(已修 9 个,新增 2 个待修)。详见下文。

### 本测试期间新发现、待修的异常(2 个)

| # | 异常 | 根因(初步) | 影响 | 建议 |
|---|------|------------|------|------|
| **10** | **重连死循环占满 worker 槽位** | `reconnect liveness probe failed, attempt anyway` —— 流已断的频道,重连前存活探测失败但代码**仍强行 attempt**,陷入"重连→失败→再重连"循环,占着 worker 槽不释放,阻塞 pending 任务 | 严重:3 个死循环频道占满 worker.num=3 槽 → 4 个新开播 pending 永远排不到 → **漏录**。在 412 风控期 / CDN 不稳时高发 | 重连策略需修正:liveness probe 失败应累计预算,耗尽即放弃,而非 attempt anyway |
| **11** | **健康检测对"0 字节文件"有盲点** | `checkRecordingHealth` 监控文件大小增长,但 ffmpeg 起来后**从未写入数据**(selectStream 拿不到流)时文件保持 0 字节,健康检测**不触发 unhealthy**(可能假设文件已有初始大小才开始监测) | 中等:僵尸 task(running 但 0 字节)无法被健康检测清理,需重启才能释放槽位 | 健康检测应覆盖"0 字节超过 N 秒"场景,或起录后 selectStream 失败应快速失败而非挂起 |

## 二、本次测试期间额外遭遇的环境问题(非代码缺陷)

### HTTP 412 IP 级风控
- **现象**:重启 hikami 后,所有 CheckLive 返回 `bilibili http status 412`(HTTP 网关层,非业务码 -352)
- **根因**:本会话**多次重启 hikami**(部署修复、清僵尸)+ 频繁请求,触发 B站对出口 IP 的网关级风控。412 比 -352 更狠:HTTP 层直接挡,不返回业务体,异常 #9 的 -352 重试逻辑不覆盖它
- **解决**:**换 IP**(重启路由器,45.66.218.55 → 38.47.123.209)→ 412 立刻解除,CheckLive 全部成功
- **结论**:412 是 IP 维度风控,正常生产不会频繁触发(本次是调试诱发的极端情况)。但暴露了**异常 #9 修复不覆盖 HTTP 层风控码(412/403 等)**的盲点 —— 已记入异常 #11 的延伸考量
- **预防**:生产环境避免短时间内频繁重启;可考虑把 412 也纳入风控冷却(同 -352 处理范式)

## 四、修复验证的关键结论

### 1. 异常 #9(-352 频率风控)— 充分验证
部署后 11 小时实战(09:12~20:09),验证了完整闭环:
- **风控冲击期**(09:12~13:00):B站延续风控,-352 脉冲每 8-9min 一波 → 三层修复(去重/冷却/jitter)兜住,**95% 成功率**,3 路录制正常起录
- **脉冲持续期**(10:13~13:00):两路录制健康完成(2h44m / 2h53m 自然下播)
- **风控解除期**(13:00~):脉冲消失,25/25 频道全部恢复正常探测
- **晚间高峰**(16:51~20:09):6 路开播全部被捕获,ERROR=0

### 2. 8 异常修复 — 整体有效,边缘未覆盖
- 异常 #1/#2/#4/#5/#6/#7/#8 均按设计工作
- 但**异常 #1/#2(流断重连)在 412 风控期 + CDN 不稳时仍会概率性失败**,表现为异常 #10(重连死循环)
- **异常 #4(健康检测)对 0 字节文件有盲点**(异常 #11)

### 3. 核心指标(全测试期)
- ERROR 日志:极少(主要在 412 期间)
- 自然下播完整保留:5+ 路(1h38m / 2h44m / 2h53m 等)
- 开播捕获率:晚间高峰 6/6(异常 #9 修复后)
- worker 池死锁:异常 #6 修复后未再出现 pending 永久阻塞(除非异常 #10 重连死循环诱发)

## 五、未解决问题的优先级与建议

### P0(高,影响核心功能)— 异常 #10:重连死循环
- **问题**:`reconnect liveness probe failed, attempt anyway` 导致流已断的频道陷入重连循环,占满 worker 槽,阻塞新开播
- **修复方向**:liveness probe 失败应累计预算(如 maxReconnect 的子集),耗尽即放弃并标记 failed,释放槽位。当前"attempt anyway"过于乐观
- **复现**:412/CDN 不稳时高发,正常网络下偶发
- **关联**:与异常 #1/#2 同源(重连策略),但表现更严重(占槽 vs 单路失败)

### P1(中,影响可观测性)— 异常 #11:健康检测 0 字节盲点
- **问题**:ffmpeg 起来但从未写入数据(selectStream 拿不到流)时,文件 0 字节,健康检测不触发 unhealthy
- **修复方向**:健康检测覆盖"起录后 N 秒文件仍 0 字节"场景;或 HandleTask 在 selectStream 失败时快速失败(不进入无限重连)
- **关联**:与异常 #10 互为表里(#10 是起录后重连死循环,#11 是起录失败后挂起)

### P2(低,防御性)— HTTP 层风控码(412/403)纳入冷却
- **问题**:异常 #9 只处理业务码 -352,HTTP 层 412/403 不触发冷却
- **修复方向**:`getJSON` 检测 HTTP 412/403 时,返回可识别哨兵(如 `ErrHTTPRiskControl`),checkOne 据此触发同款阶梯冷却
- **触发条件**:正常生产罕见(需频繁重启/异常请求模式才会诱发 IP 级风控),优先级低

### 非缺陷的"失败"显示优化(前端)
- **问题**:回顾页面把所有 failed 都显示为红色失败,但其中"重启打断"(15)和"开播即下播"(4)其实不是缺陷
- **建议**:前端按 last_error 分类标记 —— 重启打断/竞态下播显示为灰色"已跳过"而非红色"失败";或后端用更轻的状态(cancelled/skipped)

## 六、本次测试的方法学沉淀

### 有效的做法
1. **systematic-debugging 纪律**:不猜测、不扩大范围,先复现 + 取证再修。异常 #9 定位"纯频率问题而非认证问题"就是靠 identify 单次调正常这个反证
2. **TDD + codex 双审**:每个异常先写测试再实现,计划/代码各过 codex。异常 #9 计划审了 3 轮(v1→v2→v3),代码审了 2 轮,捕获了多个设计缺陷(RefreshKeys 刷错 signer / Check 绕过冷却 等)
3. **分阶段部署验证**:每批修复后部署 + 实战观察,确认有效再修下一批
4. **用证据说话**:所有结论都有日志/DB 取证,不凭感觉

### 教训
1. **频繁重启会诱发风控**:本会话多次重启 hikami 触发了 412 IP 风控,污染了测试环境。生产部署应避免短时间密集重启
2. **健康检测要覆盖边缘态**:0 字节文件、僵尸 ffmpeg 等"从未正常开始"的状态容易被健康检测忽略
3. **重连策略的乐观主义要克制**:`attempt anyway` 在流确实断了时会变成死循环,需要有预算约束
