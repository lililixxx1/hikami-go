## v3 修订说明(针对 Codex r5 的 1 Critical + 2 High + 3 Medium + 1 Low)

v2 的 `(bool, error)` helper 接口有根本缺陷:三态用二元表达,导致 `AutoReconnect=false` 时吞掉录制错误(Critical)、无法表达"耗尽但失败"、调用方易误用。v3 接受 r5 全部建议,**用显式枚举重构决策类型,并彻底重构循环结构**。

| r5 issue | v3 处理 |
|---------|---------|
| Critical:`!AutoReconnect` 吞录制错误(`return false,nil` → err 被清成 nil) | helper 改为返回**三态枚举** `afterRecordDecision` + carryErr,调用方按枚举分支处理,类型层面杜绝吞错误 |
| High 1:1c+1d 重复探测首段 | 删除 1c 独立调用,**唯一探测点在循环顶部**,首段结果与重连段结果都从这里进入判定 |
| High 2:`for attempt<max` 在最后一次重连录完后不再判定 | 用 `attemptsUsed` 计数器解耦「判定」与「执行重连」,每次 recordAudio 后先判定再决定是否重连,耗尽在判定后单独处理 |
| Medium 1:canceled 守卫内移削弱立即退出 | 循环顶部先查 `runCtx.Err()`;delay 的 `<-runCtx.Done()` 分支直接 `break`,不依赖下一轮 helper |
| Medium 2:`(bool,error)` 不自洽 | 改枚举 `afterRecordFinishSuccess / afterRecordFinishError / afterRecordReconnect` |
| Medium 3:测试 D 的 recorder 不符目标 | 改用 always-failing recorder + 补"clean EOF 仍 live 且耗尽"测试 |
| Low:`liveRecordPayload` 类型名错 | 改为 `taskPayload`(manager.go:72) |

## 一、要解决的代码缺口

`internal/live_record/manager.go:583-585`:

```go
err = m.recordAudio(runCtx, stream, audioPath, reportRecording)   // ffmpeg 干净退出时返回 nil
for attempt := 0; err != nil && ... ; attempt++ {                  // ← err != nil 守卫把 nil 放行
    ...                                                             // 重连逻辑
}
```

`RecordWithProcessStart`(manager.go:862-872)在 ffmpeg 退出码 0 + `io.Copy` EOF 返回 nil 时返回 `nil`,与"主播正常下播"无法区分。**这个缺口不只存在于首段**——循环内 `manager.go:630` 的 `recordAudio(segmentPath)` 若也返回 nil(clean EOF),`for` 的 `err != nil` 条件同样会让循环直接退出。21:52 漏录正是首段 clean EOF 被放行;修复必须覆盖**每次** recordAudio。

## 二、设计原则

1. **复用,不重写**。循环内已有「延迟 → selectStream → 分段录制」链路,只重构决策与守卫部分,保留全部副作用。
2. **判定权从 ffmpeg 退出码转移到 B 站直播状态**。以直播间状态为准。
3. **两条路径的探测错策略必须不同**(r4 Medium 2):
   - 已失败路径(`wantErr != nil`):探测错→保守重连(沿用现状)
   - 干净退出路径(`wantErr == nil`):探测错→保守收尾(保留已录音频)
4. **三态枚举表达决策**(r5 Critical/Medium 2)。决策类型在类型层面区分「正常收尾 / 保留错误退出 / 重连」,杜绝 `(bool,error)` 吞错误。
5. **可关闭**。`AutoReconnect=false` 时 helper 直接 `FinishError(wantErr)`,保留原 err 语义,零回归。

### 改动 1:`internal/live_record/manager.go`

#### 1a. 哨兵变量 + 决策枚举(manager.go:32-37 的 `var(...)` 块,及上方)

```go
// errStreamEndedWhileLive 表示 ffmpeg 收到上游 EOF 干净退出(recordAudio 返回 nil),
// 但直播间探测显示仍在播——典型场景是 CDN 切换/网络抖动导致流中途断开。
// 用于把"成功 EOF"从"正常结束"重新归类为"需要重连的中断"。
var errStreamEndedWhileLive = errors.New("stream ended cleanly while still live")

// afterRecordDecision 是「一次 recordAudio 结束后该如何继续」的三态决策。
// 用显式枚举替代 (bool,error),避免调用方把录制错误清成 nil(r5 Critical)。
type afterRecordDecision int

const (
    // afterRecordFinishSuccess:正常收尾。调用方置 err=nil,走 finalize+normalize。
    afterRecordFinishSuccess afterRecordDecision = iota
    // afterRecordFinishError:保留错误退出。调用方保留 carryErr(取消/耗尽/原录制失败)。
    afterRecordFinishError
    // afterRecordReconnect:进入或继续重连。
    afterRecordReconnect
)
```

#### 1b. 决策 helper(recordAudio/manager.go:686 附近)

```go
// decideAfterRecord 判定一次录制结束后该如何继续。wantErr 是本次 recordAudio 的返回值
// (nil 表示 ffmpeg 干净退出,非 nil 表示拉流/进程/拷贝失败)。
//
// 返回 (decision, carryErr):
//   - FinishSuccess:调用方 err = nil 正常收尾(下播,或干净退出+探测错保守收尾)
//   - FinishError  :调用方 err = carryErr 退出(canceled、AutoReconnect=false 且 wantErr!=nil)
//   - Reconnect    :调用方继续重连(仍开播,或已失败路径+探测错保守重连)
//
// 探测错兜底方向因路径而异(设计原则 3):
//   wantErr != nil:反正要重连,探测错仍重连(沿用 manager.go:603-610);
//   wantErr == nil:本可收尾,探测错保守收尾(避免丢弃已录音频)。
func (m *Manager) decideAfterRecord(ctx context.Context, task worker.Task, payload taskPayload, wantErr error) (afterRecordDecision, error) {
    // 关闭重连:保留原 err 语义,不吞错误(r5 Critical)
    if !m.cfg.LiveRecord.AutoReconnect {
        if wantErr == nil {
            return afterRecordFinishSuccess, nil
        }
        return afterRecordFinishError, wantErr
    }

    liveInfo, liveErr := m.client.CheckLive(ctx, payload.RoomID, m.cookieHeaderForChannel(ctx, task.ChannelID))

    // 明确已下播:正常收尾(无论 wantErr 是 nil 还是错——下播即结束)
    if liveErr == nil && !liveInfo.Live {
        slog.Info("stream ended, skipping reconnect", "channel_id", task.ChannelID, "room_id", payload.RoomID)
        return afterRecordFinishSuccess, nil
    }
    // 探测期间被取消:保留错误退出,交外层取消收尾
    if liveErr != nil && errors.Is(ctx.Err(), context.Canceled) {
        return afterRecordFinishError, wantErr
    }

    switch {
    case liveErr == nil && liveInfo.Live:
        // 明确仍开播:重连。wantErr==nil(干净 EOF)时用哨兵触发。
        carry := wantErr
        if carry == nil {
            slog.Warn("stream ended cleanly while still live, attempting reconnect",
                "channel_id", task.ChannelID, "room_id", payload.RoomID)
            carry = errStreamEndedWhileLive
        }
        return afterRecordReconnect, carry
    default:
        // liveErr != nil(探测出错):
        //   wantErr != nil → 保守重连(现状语义)
        //   wantErr == nil → 保守收尾(不丢弃已录音频)
        if wantErr == nil {
            slog.Info("clean exit with inconclusive live probe, treating as ended",
                "channel_id", task.ChannelID, "room_id", payload.RoomID, "probe_error", liveErr)
            return afterRecordFinishSuccess, nil
        }
        slog.Warn("reconnect liveness probe failed, attempt anyway",
            "channel_id", task.ChannelID, "room_id", payload.RoomID, "probe_error", liveErr, "stream_error", wantErr)
        return afterRecordReconnect, wantErr
    }
}
```

> **接口约束自洽性**:三态下 carryErr 的含义清晰——`FinishSuccess` 时调用方忽略 carryErr(置 nil);`FinishError` 时 carryErr 必非 nil(取消场景 wantErr 可能是 nil,但 ctx.Err() 已 canceled,外层收尾靠 ctx 判定);`Reconnect` 时 carryErr 是原 wantErr 或哨兵。`(Reconnect, nil)` 不可能出现(live 分支已保证 carry 非 nil)。

#### 1c. 重构循环(manager.go:583-632,这是核心改动)

**唯一探测点在循环顶部**,首段与重连段结果都从这里进入判定;**`attemptsUsed` 计数器**解耦判定与重连执行(r5 High 2/3)。

```go
err = m.recordAudio(runCtx, stream, audioPath, reportRecording)
addAudioSegment(audioPath)

attemptsUsed := 0
for {
    // 循环顶部:唯一探测点。判定「上一次 recordAudio 的结果(首段或重连段)该如何继续」。
    // 这覆盖了 r4 High 1:每次 recordAudio 后(含重连分段)都过 helper。
    decision, carryErr := m.decideAfterRecord(runCtx, task, payload, err)

    // 取消优先:Stop/worker 取消发生时立即退出,不因 helper 内的 CheckLive 而延迟(r5 Medium 1)
    if errors.Is(runCtx.Err(), context.Canceled) {
        err = carryErr
        if err == nil {
            err = runCtx.Err() // canceled 但 recordAudio 成功的场景,保留取消原因
        }
        break
    }

    switch decision {
    case afterRecordFinishSuccess:
        err = nil
        break
    case afterRecordFinishError:
        err = carryErr
        break
    case afterRecordReconnect:
        // 想重连但额度耗尽:退出并保留错误。clean-EOF+仍 live 耗尽时返回哨兵(r5 Medium 3 测试目标)。
        if attemptsUsed >= maxReconnect {
            err = carryErr
            break
        }
        err = carryErr
        _ = reporter.Progress(ctx, 15+attemptsUsed*5, fmt.Sprintf("reconnecting (attempt %d/%d)", attemptsUsed+1, maxReconnect))

        // 可取消的等待:runCtx.Done() 时直接 break 到外层取消收尾(r5 Medium 1)
        select {
        case <-time.After(reconnectDelay):
        case <-runCtx.Done():
            err = runCtx.Err()
            break
        }

        stream, sErr := m.selectStream(runCtx, payload.RoomID, m.cookieHeaderForChannel(ctx, task.ChannelID))
        if sErr != nil {
            slog.Warn("reconnect stream selection failed", "error", sErr)
            err = sErr
            attemptsUsed++ // 选流失败也算一次重连尝试(沿用现状:continue 后 for 条件 attempt++)
            continue
        }

        segmentPath := reconnectAudioSegmentPath(audioPath, attemptsUsed+1)
        segErr := m.recordAudio(runCtx, stream, segmentPath, reportRecording)
        addAudioSegment(segmentPath)
        err = segErr
        attemptsUsed++
        continue
    }
    break
}
```

> **Go 注意**:`switch` 内的 `break` 只跳出 switch。需用带标签 break 或把 switch 改写为 if/else,确保 decision 分支能退出 for。实现时用 `if/else if/else` 结构而非 switch,或加 `decided:` 标签。这是实现细节,不影响设计。

#### 1d. 收尾分支不变

`manager.go:634-684` 的收尾逻辑(`if err != nil` 取消分支 / finalize / UpdateEndedAt / Apply / enqueueNormalize)**完全不动**。新循环产生的 `err`(nil / 哨兵 / 原错误 / canceled)自然流回这些分支。

```go
// statefulLiveClient 按调用次数返回预设的开播探测结果,精确驱动各阶段(r5 High 2/3)。
type statefulLiveClient struct {
    lives []bool      // 每次 CheckLive 期望返回的 live 值
    errs  []error     // 可选:对应次数的探测错误(非 nil 时优先)
    calls int
}
func (c *statefulLiveClient) CheckLive(ctx context.Context, roomID int64, cookieHeader string) (LiveInfo, error) {
    n := c.calls; c.calls++
    if n < len(c.errs) && c.errs[n] != nil {
        return LiveInfo{}, c.errs[n]
    }
    live := false
    if n < len(c.lives) { live = c.lives[n] }
    return LiveInfo{RoomID: roomID, Live: live, Title: "stateful", StartedAt: time.Date(2026,4,27,13,0,0,0,time.Local)}, nil
}
func (c *statefulLiveClient) GetStream(ctx context.Context, roomID int64, audioOnly bool, cookieHeader string) (StreamInfo, error) {
    return StreamInfo{URL: "https://example.com/live.flv", AudioOnly: true}, nil
}

// cleanEOFRecorder 每次 record 都返回 nil(模拟 ffmpeg 干净 EOF),写有效字节。
type cleanEOFRecorder struct{ outputs []string }
func (r *cleanEOFRecorder) Record(ctx context.Context, stream StreamInfo, outputPath string) error {
    r.outputs = append(r.outputs, outputPath)
    return os.WriteFile(outputPath, []byte("segment-"+strconv.Itoa(len(r.outputs))+"\n"), 0o644)
}

// alwaysFailingRecorder 模拟持续拉流/录制失败(r5 Medium 3)。
type alwaysFailingRecorder struct{ outputs []string }
func (r *alwaysFailingRecorder) Record(ctx context.Context, stream StreamInfo, outputPath string) error {
    r.outputs = append(r.outputs, outputPath)
    return errors.New("stream interrupted")
}
```
