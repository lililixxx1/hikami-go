# 回顾生成系统优化方案（v3）

> 基于 26.04.22 场次对比分析 + DeepSeek API 官方文档验证 + 实测数据

## 核心发现

### 官方文档参数（api-docs.deepseek.com）

| 参数 | deepseek-v4-flash | deepseek-v4-pro |
|------|-------------------|-----------------|
| 上下文长度 | **1M tokens** | **1M tokens** |
| 最大输出 | **384K tokens** | **384K tokens** |
| 定价（输出） | $0.28/1M tokens | $0.87/1M tokens（75%折扣后） |
| 并发限制 | 2500 | 500 |

### 实测数据

| 模型 | max_tokens 设置 | 实际 completion_tokens | 实际输出字符 | 备注 |
|------|----------------|----------------------|-------------|------|
| v4-flash | 16384 | 5,501 | 9,831 字 | 通用测试 |
| v4-flash | 32768 | 5,898 | 8,822 字 | 加大 max_tokens 无效 |
| v4-pro | 32768 | 7,744 | 12,486 字 | 通用测试 |
| v4-flash | 16384 | 3,841 | 12,9 KB | **26.04.22 回顾实测** |
| **v4-pro** | **16384** | **~7000** | **20.4 KB** | **26.04.22 回顾实测** |

**v4-pro 在 26.04.22 实际回顾任务中输出 20.4KB，超过原文稿的 18.3KB。**

v4-pro 回顾质量评估：
- ✅ 术语标注 `[应为：...]` 残留：**0 处**（v4-pro 自行清理了）
- ✅ 分段数：**9 段**（接近 prompt 要求的 8-10 段）
- ✅ 故事细节：保留了撕明信片、午托班、许留山等关键情节
- ⚠️ ASR 错误残留：1 处（"灰烬本"出现在正文中，非引用块）

### 根因分析

**API 上限不是瓶颈（384K 远超需求）。** 两个模型都在 5500-8000 completion_tokens 时自然停止（finish_reason: stop），这是**模型自身的"写作意愿"上限**——它认为"写够了"就停了。

- v4-pro 比 v4-flash 多输出约 **40%**（7744 vs 5501 tokens）
- 3841 tokens 的回顾（新文稿）远低于模型能力，说明 prompt 质量是关键
- 原文稿 18KB 用的是更早期的模型/配置，可能当时模型行为不同

### 解决方向

1. **换 v4-pro** — 立即提升输出长度 40%，预计可达 18KB 级别
2. **分段生成** — 彻底突破单次调用上限，可生成任意长度的回顾
3. **Prompt 工程优化** — 引导模型写得更详细、更充分

## DeepSeek 模型自身分析结论

直接向 DeepSeek V4-Pro 提问分析此问题，核心结论：

1. **`finish_reason: stop` = 模型主动停止，不是 API 截断**。384K 上限是机械层面支持，实际生成长度由模型判断"任务完成"决定。
2. **模型的"惰性生成"行为**：训练语料中高质量回答通常较短，模型学到了"用户不需要几万字回答"。
3. **突破方法（按有效性排序）**：
   - 明确量化字数要求："总长度不少于 15000 tokens，在达到前不能停止"
   - 强制结构/逐章节："每章不少于 800 字，完成所有章节前不许停止"
   - 禁止概括："宁可保留口水话也不要省略细节"
   - 自检指令："如果快结束了，检查是否达到要求长度，没达到就继续写"
   - 自动续写：`finish_reason: stop` 且长度不足时，发起新请求续写
4. **Reasoning tokens 影响**：thinking 消耗 max_tokens 配额，但当前场景影响不大（481/5253）
5. **Stream 不增加输出长度**，但保障长生成的连接稳定性

## 一、问题与方案

### P0: 模型单次输出"写作意愿"上限 — 核心瓶颈

**现象：** v4-flash 单次输出 ~5500 tokens，v4-pro ~7700 tokens，即使 max_tokens 设 32K 也不会写更多。

**方案 A（✅ 已验证）：切换到 v4-pro**
- 直接改配置 `model: "deepseek-v4-pro"`
- 实测输出 20.4KB（超过原文稿 18.3KB）✅
- 术语标注残留 0 处 ✅
- 分段 9 段（接近目标 8-10 段）✅
- 成本增加 3.1x（$0.87 vs $0.28/1M output tokens），绝对值极低
- 耗时增加：2m17s vs 57s（可接受）

**方案 B（推荐，根本解决）：分段生成+合并**

**现象：** 回顾只能输出 ~4000 tokens（~13KB），无法达到原文稿 18KB 的详细程度。

**方案 A（推荐）：分段生成+合并**
1. 使用 `segmentation.go` 的分段结果，将直播按话题拆为 3-5 个片段
2. 每个片段独立调用 API 生成回顾段落（每段 ~2000 tokens，在上限内）
3. 最后一次 API 调用合并所有段落，生成概要/高光/语录/建议/结尾
4. 总输出可达 5 段 × 2000 + 汇总 2000 = ~12000 tokens（~40KB），远超当前

**方案 B：升级到 deepseek-v4-pro**
- 如果 v4-pro 有更大的 `max_output_tokens`（如 16K-32K），可直接解决
- 需验证 pro 模型的输出上限
- 成本更高但实施最简单

**方案 C：切换 Provider**
- 使用输出上限更高的模型（如 Claude，200K 输出）
- 项目已支持 anthropic provider，切换成本低

### P0: 术语标注 `[应为：...]` 残留

**现象：** 最终 Markdown 出现 `[应为：做题龙]`、`[应为：灰泽满]` 等标记。

**原因：** prompt 指令要求原话引用中保留 `[应为：xxx]` 标注，`extractSuggestedTerms` 提取后没有从正文移除。

**修复：** `glossary_correction.go` 的 `applyGlossaryCorrections` 末尾增加一步：
```go
// 提取术语建议后，清除所有标注残留
result = regexp.MustCompile(`\[应为[：:][^\]]+\]`).ReplaceAllString(result, "")
```

### P0: ASR 错误原样出现在正文

**现象：** "灰烬本"、"灰子版"、"伊丽莎" 出现在回顾正文。

**修复：**
1. `applyGlossaryCorrections` 增加全文 ASR 错误→正确术语替换（复用 glossary term→canonical 映射）
2. 不局限于 `[应为：]` 标记，直接全文搜索替换
3. 注意：原话引用（`>` 块）中保留原始说法但移除标注标记

### P1: Prompt 分层优化（配合分段生成）

**现状：** 198KB prompt 一次性发送，AI 要在 8K 输出限制内覆盖 3 小时 41 分钟内容。

**优化：**
1. **分段 prompt**：每段只发送该段的 transcript + 弹幕 + 上下文摘要
2. **弹幕精简**：58250 条弹幕的全文注入过多，只保留：
   - 各峰值窗口 top 5 代表性弹幕
   - 术语相关的弹幕
   - 情感极值弹幕（最活跃/最安静时刻）
3. **上下文传递**：前一段的摘要作为后一段的 context，保持连贯性

### P1: 叙事段落分段增强

**现象：** 40 分钟故事被压缩为一段。

**修复：**
1. 检测长时间低弹幕密度区间（<5 条/分钟持续 >10 分钟），按时间均匀分段
2. prompt 增加指令："叙事性内容按情节转折分段"
3. 与分段生成配合，每段故事独立生成

### P2: 时间戳格式统一

**修复：** 后处理正则 `\d{2}:\d{2}:\d{2}` → `MMM:SS`，或 prompt 中更强约束。

## 二、DeepSeek 官方最佳实践（模型自述）

1. **分块策略**：将长文本拆为小段，按序生成，保持平滑过渡
2. **上下文摘要**：每段生成前注入前文摘要，维持连贯性
3. **递归摘要**：周期性总结已生成内容，注入后续 prompt
4. **渐进式生成**：生成固定长度片段，检查逻辑一致性后继续
5. **结构化 prompt**：使用大纲/章节标题引导模型，保持主题一致
6. **监控输入长度**：修剪过旧历史，确保不超上下文窗口

## 四、分段生成架构设计

```
                  ┌─────────────────┐
                  │   HandleTask    │
                  └────────┬────────┘
                           │
                  ┌────────▼────────┐
                  │ 判断是否需要分段  │
                  │ duration > 90min │
                  │ 或 transcript 大 │
                  │ 于阈值           │
                  └───┬─────────┬───┘
                      │         │
               不分段  │         │  需要分段
                      │         │
               ┌──────▼──┐  ┌───▼────────────┐
               │ 单次生成 │  │ Segmentation   │
               │ (现有)  │  │ 分为 N 段       │
               └──────┬──┘  └───┬────────┬───┘
                      │        │        │

                      │   └───┬───┘ └──┬───┘
                      │       │        │
```
