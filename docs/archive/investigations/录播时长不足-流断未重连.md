# 录播时长不足:直播流断后未重连

> **状态**:根因已更正 + **已修复实施**(2026-07-04,经 codex-review 7 轮复核,见 `docs/archive/investigations/录播漏录-修复计划.md` 与 `reviews/main--r41.md`)
> **发现日期**:2026-07-04
> **影响场次**:`bili_1298779265_live_20260703_211438`(2026-07-03 晚灰泽满直播)
> **严重程度**:高(直播内容永久丢失,无法补救)
>
> **勘误说明**:本文档初版曾断言根因为「`auto_reconnect` 无默认值 → 默认 false → 不重连」。
> 经核对当前源码与同场日志,该结论**不成立**(`auto_reconnect` 自初始提交起默认 `true`,
> 同场日志 21:15 也出现了 `attempt:1/3`、`attempt:2/3` 的重连记录)。
> 真实根因见下方「四、根因分析」。保留勘误说明以防再次踩坑。

## 一、现象

用户报告:2026-07-03 晚主播灰泽满的直播实际从 21:14 持续到次日 00:28(约 3 小时 14 分),但 Hikami 只录到了 **37 分钟**(`21:15–21:52`),生成的回顾文档基于这段不完整的录音,内容严重缺失。用户怀疑是"打开了自动上传导致录播停止"。

## 二、结论速览

**不是自动上传导致的。** 自动上传发生在录制之后的 recap 阶段(本场上传实际在次日 07:13,且只读已录好的文件,不可能停止录制)。

**也不是「重连未启用」。** `auto_reconnect` 在 `internal/config/config.go:750-752` 自初始提交起就默认为 `true`(`max_reconnect=3`、`reconnect_delay_seconds=10`),同场日志 21:15 也确实记录了两次 `stream interrupted, attempting reconnect`(attempt 1/3、2/3)。

**真正根因:ffmpeg 收到上游 EOF 以 `status:success`(退出码 0)正常退出时,`FFmpegRecorder.RecordWithProcessStart` 返回 `nil`,而外层重连循环仅在 `err != nil` 时执行——于是「直播流中断但 ffmpeg 干净退出」这种最常见场景反而绕过了重连**,即便 `AutoReconnect=true` 也不会重连。21:52 正是这种情况:流断了,ffmpeg 收到 EOF 正常退出,录制直接 `finished`,而主播实际仍在播(持续到 00:28),导致 **21:52 → 00:28 约 2.5 小时的直播内容完全未录到**。

## 三、证据链

### 3.1 录制在 21:52 停止(ffmpeg 干净退出,非被杀)

`logs/hikami-20260703-084908.log` 关键行(实际时间戳与 JSON 字段,有删节):

```
21:15:08  live record start requested
21:15:09  ffmpeg record started   args[0]=-y            (首次启动,output=audio.m4a)
21:15:13  WARN stream interrupted, attempting reconnect  attempt=1/3  error="http status 404"
21:15:23  ffmpeg record started   args[0]=-n            (重连第1次,output=audio.part.1.m4a)
21:15:27  WARN stream interrupted, attempting reconnect  attempt=2/3  error="http status 404"
21:15:37  ffmpeg record started   args[0]=-n            (重连第2次,output=audio.part.2.m4a,之后稳定 37 分钟)
21:52:38  ffmpeg process exited   status=success        ← 流断了,ffmpeg 收到 EOF 正常退出(退出码 0)
21:52:38  live record finished    duration=37m29s file_size=52442139
```

- **21:15 的 3 次 `ffmpeg record started` 不是"流握手",而是 1 次首启 + 2 次自动重连**(中间夹着两条 `stream interrupted, attempting reconnect` WARN,`attempt=1/3`、`2/3`)。这证明运行时 `AutoReconnect=true`、`max_reconnect=3` 已生效。初版文档把它们描述成"流握手重试"是误读。
- `attempt=3/3` 在 21:52 之后再未出现 —— 因为 21:52 的退出被当作"成功结束",根本没进入第 3 次重连(见 §3.3)。
- 21:52:38 ffmpeg 以 `status: success` 退出 —— 即 ffmpeg 进程退出码为 0(收到上游 EOF 正常收尾),不是被 SIGTERM/SIGKILL。

### 3.2 直播实际仍在继续(主播没下播)

21:52 之后,scheduler 持续轮询 B站直播间状态,API 在主播下播前**始终回报 `live: true`**(精确统计:`logs/hikami-20260703-084908.log` 中 `room_id=1713546334` 共 **934 条 `live:true`**):

```
21:53:08  bilibili live status checked  live=true  title=悠闲夜晚  ← 21:52 后首条 live:true
...       (21:53 → 00:28,约 2 小时 35 分,934 条 live:true,0 条 live:false)
00:28:08  bilibili live status checked  live=true  title=悠闲夜晚  ← 最后一条 live:true
00:28:38  bilibili live status checked  live=false                  ← 首次 live:false(主播下播)
```

**主播实际在 2026-07-04 00:28 才下播**(与用户"凌晨1点多"的表述基本吻合),比录制停止时刻(21:52)晚了 **2 小时 36 分**。这 2.5 小时 scheduler 探测到 934 次"仍在播",但未触发任何重录(原因见 §3.3 末段)。

### 3.3 21:52 后系统未尝试重新录制(成功退出短路了重连循环)

整份日志在 21:52:38 之后,**该 session 再无任何 `ffmpeg record started` / `stream interrupted, attempting reconnect` 日志**。即便 scheduler 在 21:52–00:28 期间持续探测到 `live: true`,也没有触发重新录制。

关键在于 `internal/live_record/manager.go:583-585` 的控制流:

```go
err = m.recordAudio(runCtx, stream, audioPath, reportRecording)   // ← 21:52 这一行
// ↓ 循环条件:err != nil。21:52 ffmpeg status=success → recordAudio 返回 nil → 循环体一次都不执行
for attempt := 0; err != nil && !errors.Is(runCtx.Err(), context.Canceled) && attempt < maxReconnect; attempt++ {
    ...
}
```

而 `recordAudio` → `FFmpegRecorder.RecordWithProcessStart`(manager.go:814-873)在 ffmpeg 正常退出时返回 `nil`:

```go
waitErr := cmd.Wait()                       // ffmpeg 退出码 0 → waitErr == nil
...
if waitErr != nil { return ... }            // 不进
if copyErr != nil && ctx.Err() == nil { return ... }  // io.Copy 因 EOF 返回 nil → 不进
return nil                                  // ← 21:52 走这里
```

也就是说:**「直播流中途断了,但 ffmpeg 收到 EOF 干净退出」这种最常见、最该重连的场景,被重连循环的 `err != nil` 守卫直接放行了**。重连逻辑只能捕获"ffmpeg 进程崩溃/非零退出"或"流拷贝报错"两类失败,捕获不到"流正常结束"。

> 这也解释了为什么 21:15 的两次重连能触发(那时是 `open live stream: http status 404`,拉流阶段就报错,`err != nil`),而 21:52 却不能(ffmpeg 已正常退出)。

**第二道闸门:scheduler 不会对已在流水线中的 session 重录。** 21:52 录制"成功结束"后,系统立即把它当作完整场次推进下游:`21:53:00 auto ASR submitted` → `21:55 asr completed` → `21:55 auto recap submitted`。scheduler 的 `live_check` 轮询职责是"发现新开播并首发录制",不负责"对已结束 session 复录"——于是 21:53→00:28 期间 934 次 `live:true` 全部落空,没有任何 `live record start` 被发起。这意味着:**只要 manager 在断流瞬间没接住(第一道闸门失守),后续没有任何兜底会把这 2.5 小时找回来**。这是方案 A(manager 层 EOF 后探测重连)必须落地的核心理由——它是在"session 被认定为结束"之前唯一的补救点。

## 四、根因分析

### 4.1 重连机制代码(`internal/live_record/manager.go:570-632`)

```go
// Recording with optional auto-reconnect
maxReconnect := 0
reconnectDelay := 10 * time.Second
if m.cfg.LiveRecord.AutoReconnect {                        // 运行时为 true(默认)
    maxReconnect = m.cfg.LiveRecord.MaxReconnect           // 默认 3
    if maxReconnect <= 0 { maxReconnect = 3 }
    if m.cfg.LiveRecord.ReconnectDelay > 0 {
        reconnectDelay = time.Duration(m.cfg.LiveRecord.ReconnectDelay) * time.Second
    }
}

err = m.recordAudio(runCtx, stream, audioPath, reportRecording)
// ↓ 关键守卫:err != nil。21:52 ffmpeg status=success → recordAudio 返回 nil → 循环不执行
for attempt := 0; err != nil && !errors.Is(runCtx.Err(), context.Canceled) && attempt < maxReconnect; attempt++ {
    // 重连前的存活探测:明确判定下播(!Live)才放弃;仍在播则重连
    liveInfo, liveErr := m.client.CheckLive(...)
    if liveErr == nil && !liveInfo.Live { break }
    // ... selectStream + recordAudio 重新拉流录制
}
```

重连循环内部的存活探测设计得很健壮(下播才放弃、仍在播则重连、容忍探测出错)。**但它有一个前置守卫:`err != nil`。这个守卫把"流正常结束"和"流异常中断但 ffmpeg 干净退出"一并放行了,绕过了整段重连逻辑。**

### 4.2 真正根因:成功 EOF 短路了重连(`manager.go:814-873`)

`FFmpegRecorder.RecordWithProcessStart` 的返回逻辑:

```go
waitErr := cmd.Wait()                                  // manager.go:862
if waitErr != nil { return fmt.Errorf("ffmpeg record failed: ...") }  // :866-868
if copyErr != nil && ctx.Err() == nil { return fmt.Errorf("copy ...") } // :869-871
return nil                                             // :872 ← 21:52 走这里
```

直播流中断时,上游关闭 socket,`io.Copy` 收到 EOF 返回 `nil`,`cmd.Wait()` 看到 ffmpeg 退出码 0 —— 三个条件全部满足,函数返回 `nil`。**对调用方而言,这和"主播正常下播、录制圆满结束"完全无法区分。**

于是:直播流断了 → ffmpeg 干净退出 → `recordAudio` 返回 nil → `for ... err != nil ...` 不执行 → 直接 `live record finished`,而主播其实还在播。**这就是 21:52 漏录的根因。**

> 反例印证:21:15 的两次 404 发生在**拉流阶段**(`open live stream: http status 404`,ffmpeg 还没正常跑起来就报错),所以 `err != nil` 成立,顺利进入重连。两相对比说明:重连逻辑只覆盖"拉流/进程失败",覆盖不到"流中途 EOF"。

## 五、影响评估

- **本次**:7/3 晚灰泽满直播 21:52–00:28 的 2.5 小时内容永久丢失(音频、弹幕均无)。
- **历史**:凡是直播中途流断、且 ffmpeg 收到干净 EOF(退出码 0)的场次,都会在断流瞬间结束录制——即使 `auto_reconnect=true` 也无济于事(见 §4.2)。回查 DB 应以「录制时长显著短于 scheduler 探测到的 live 时长 + 该场末条 ffmpeg 日志为 `status:success`」为筛选条件,而非「`auto_reconnect` 是否开启」。
- **7/2 那场经 DB 核实排除**:`bili_1298779265_live_20260702_200613` 在 `sessions` 表中 `started_at=20:06` / `ended_at=次日 01:31`,**录制时长 325 分钟(约 5.4 小时),完整覆盖**,非同类问题。(该场日志文件已不在 `logs/`,无法逐行复核,但 DB 时长足以排除漏录。)
- **触发概率**:B站直播流中断较常见(网络抖动、CDN 切换、主播端推流器重启),只要中断方式是"上游正常关闭导致 ffmpeg 干净退出",当前代码必然漏录——这是比"重连未启用"更普遍、且 `auto_reconnect` 默认开启也无法覆盖的盲区。本场的特殊性在于:21:15 的两次中断是"拉流阶段 404"(被重连接住),而 21:52 是"录制中途 EOF"(被放行)——两种中断方式在当前代码里命运截然不同。

## 六、修复方案

> **✅ 已实施(2026-07-04)**:方案 A 已落地。实现:`internal/live_record/manager.go` 新增 `decideAfterRecord` helper(三态枚举 `afterRecordFinishSuccess/FinishError/Reconnect`)+ `errStreamEndedWhileLive` 哨兵,重构主循环让**每次** recordAudio 后(含重连分段)都过 helper 判定。覆盖测试 5 个 + 修正 1 个原有测试,全量 `go test ./...` 零失败。详见 `docs/archive/investigations/录播漏录-修复计划.md`,Codex 审核报告 `reviews/main--r41.md`(r7 APPROVED)。

### 方案 A:成功 EOF 时探测「仍在直播」才判定为正常结束(治本,推荐)

问题在于 `recordAudio` 返回 `nil` 时,外层无法区分"主播正常下播"和"流中途断了但 ffmpeg 干净退出"。治本做法是**在 ffmpeg 成功退出后、决定是否重连前,加一次直播间存活探测**,把判定权从"ffmpeg 退出码"转移到"B 站直播状态"。

落地点在 `manager.go:583` 之后、`for` 循环之前。当 `err == nil && AutoReconnect` 时,主动调用一次 `m.client.CheckLive(...)`:

- 仍在播(`Live == true`)→ 伪造一个非 nil 的 `err`(如 `err = errors.New("stream ended while still live")`)让 `for` 循环接管重连;或直接复用循环体内的 `selectStream + recordAudio` 逻辑另起一段。
- 已下播(`Live == false`)→ 保持 `err = nil`,正常收尾。
- 探测本身出错 → 沿用循环体里"探测出错仍尝试重连"的保守策略(避免 B 站风控 -352 误杀)。

- 优点:从根上覆盖"流 EOF 但主播未下播"这一最常见漏录场景;复用已有存活探测语义。
- 风险:中。需注意探测频率(避免触发 B 站风控)、`CheckLive` 失败时的保守重试与现有 `max_reconnect` 上限的配合。

### 方案 B:重连上限耗尽后由 scheduler 兜底重录(第二道防线)

即便修了方案 A,连续 `max_reconnect`(默认 3)次重连失败后仍会放弃。若主播长时间直播、流反复抖动,可能再次漏录。可考虑:
- 重连上限耗尽后,若 scheduler 后续仍探测到 `live: true`,**重新发起一次完整的录制任务**(而非依赖 manager 内部循环)。这需要 scheduler 与 live_record 的协作,改动较大。

### 方案 C:文档/示例补强

`config.full.example.yaml` 当前未包含 `live_record:` 段,补 `auto_reconnect` / `max_reconnect` / `reconnect_delay_seconds` 的示例与说明,方便用户调整重连参数。注意:此项**独立于本次事故根因**(默认值已正确),仅是示例文档缺失。

> **关于初版"方案 A:补三条 SetDefault"**:经核对 `config.go:750-752` 三条 SetDefault **早已存在**,该方案是基于错误根因得出的无效建议,已删除。
