## 一、根因(探针实测确认,非猜测)

### 现象
重启 hikami(PID 131019,含 8 异常修复)后,scheduler 每 30s 触发一次 `CheckAndStartAll`,实测:
- 07:42 启动初期:79 ok / 25 -352(每分钟 ~104 次 `getInfoByRoom`)
- 07:54 起:**0 ok / 50 -352(100% 被风控)**,持续未恢复
- 同期 `POST /api/channels/identify`(单次调用)→ **200 code=0 正常**(返回 UID 1401928)

### 结论:病在频率,不在认证
`internal/live_record/bilibili.go` 的 `CheckLive` **已具备全套风控对抗**(8 异常修复时下沉):
- ✅ buvid 注入(`injectAntiRisk` → `BuvidStore.GetBuvids`,L77-90)
- ✅ WBI 签名(`signURL` → 按 cookie 缓存的 `WBISigner`,L93-100)
- ✅ 浏览器 UA + `Referer: https://live.bilibili.com/{roomID}` + `Origin`(getJSON L266-287)

identify 单次调正常证明 buvid+WBI+headers 本身有效。**纯频率问题**:25 频道 × 30s tick,且每个 live+auto 频道**启动调度路径多发** `getInfoByRoom`,突发 ~50 req/min 击穿 B站阈值。

### 调用链定位(v2 修正:补充 HandleTask preflight)

**第 1 层修复的是"启动调度路径"上的 #2(`Start` 内的二次 CheckLive)**,把它减到 1 次。完整启动链路原本 4 发(调度 2 + HandleTask 2),第 1 层后调度路径减为 1 发、完整链路减为 3 发。HandleTask 的 preflight(#3)是录制开始前的必要再确认(防窗口期内下播),**不在本计划裁剪范围**。

**零节流**:无 jitter、无频道间间隔、无 -352 感知退避。失败仅 WARN(`#8` 修复)后下一 tick 立即原节奏重打。

## 二、修复策略:三层组合

> 设计原则:**对齐本仓既有的 -352 处理范式**(`live_record/danmaku.go:326` 的单次重试 + `RefreshKeys`),不引入新依赖(`golang.org/x/time/rate` 在 go.mod 不存在)。

### 第 1 层(核心):`checkOne` 消除对 `Start` 的二次 CheckLive
**问题**:`checkOne` 已 `CheckLive` 拿到 `info.Live=true`,却调 `m.Start(ctx, item.ID)`;而 `Start`(manager.go:362-366)**再次** `CheckLive`——纯冗余。
**方案**(v3 修正:`ensureStartAllowed` 承载 `Start` 的**全部**前置防护):
- 抽 **`ensureStartAllowed(ctx, item) error`**:承载 `Start` 的**全部**前置防护(codex v2 #2),包括:
  - `!m.cfg.LiveRecord.Enabled` → `ErrLiveDisabled`(L342)
  - `item.LiveRoomID <= 0 || !item.Enabled` → `ErrChannelNotRecordable`(L350)
  - `activeFor(channelID)` 已在录 → `ErrAlreadyRecording`(L353)
  - `ActiveLiveForChannel` 已有活跃场次 → `ErrAlreadyRecording`(L356)
  - `Start` 和 `checkOne` 复用此助手
- 抽 **`startWithInfo(ctx, item, info LiveInfo) (Status, error)`**:承载 `CreateLive`(L371)+ `Enqueue`(L396)+ 返回 Status(L410)。**只负责建 session 和入队**。
- **`Start`** 改为:`ensureStartAllowed` → `CheckLive`(L362)→ `startWithInfo`。公共签名、错误语义、handler/server.go:976 调用路径**完全不变**。
- **`checkOne`** 改为:`ensureStartAllowed`(复用**全部**防护)→ 若 `ErrAlreadyRecording` 走既有兜底 → 否则 `startWithInfo(ctx, item, info)`(透传已得的 info,**省掉 #2**)。
**风险**:低。`Start` 行为等价;`checkOne → startWithInfo` 路径的防护与原 `m.Start` 完全对齐(codex v2 #2):录制禁用 / 频道不可录 / 已在录 / 已有活跃场次 全部挡在建 session 之前。

### 第 2 层(关键):对齐 danmaku.go 的 -352 单次重试 + 频道级冷却
**问题**:`bilibili.go:110` 的 `if response.Code != 0` 把 -352 与其它错误一视同仁,调度器每 30s 全量重打。

#### 2a. `bilibili.go` 的 `CheckLive`(v3 修正:signer 按 baseCookie 选 + query 闭包用同一 signer)
**关键坑**(codex v1 #2/#3 + codex v2 设计问题 #1):
- `signers` 是 `map[string]biliutil.URLSigner`(`URLSigner` 接口只有 `SignURL`,无 `RefreshKeys`)
- `injectAntiRisk` 会往 cookie 加 buvid3/buvid4 → `signURL(endpoint, injected)` 会 `signerForCookie(injected)`,**按注入后的 cookie 建 signer**;但 -352 重试刷的是 `signerForCookie(baseCookie)` —— **两个不同的 signer**(codex v2 #1 铁证)
- WBI 密钥源自 B站账号身份(nav API),**不**随 buvid 变;signer 应**按 baseCookie 选**(账号维度),`injected` 仅用作请求 Cookie 头

**方案**(v3:signer 统一按 baseCookie,根除 signer 错配):
```go
func (c *BilibiliClient) CheckLive(ctx, roomID, cookieHeader) (LiveInfo, error) {
    baseCookie := cookieHeader  // 原始入参,同时是 BuvidStore 缓存 key + signer 选择 key
    query := func() (roomInfoResponse, error) {
        // 每次都从 baseCookie 重新注入 buvid → 得 injected(仅用于 Cookie 头)
        injected := c.injectAntiRisk(ctx, baseCookie)
        endpoint := c.baseURL + "/xlive/.../getInfoByRoom?room_id=" + ...
        // signer 按 baseCookie 选(v3:不再按 injected),WBI 密钥随账号不随 buvid
        if signed, err := c.signerForCookie(baseCookie).SignURL(endpoint); err == nil {
            endpoint = signed
        }
        var resp roomInfoResponse
        err := c.getJSON(ctx, endpoint, &resp, injected, roomID)  // Cookie 头用 injected
        return resp, err
    }
    resp, err := query()
    if err != nil { return LiveInfo{}, err }
    // -352 单次重试(对齐 danmaku.go:326)
    if resp.Code == -352 {
        slog.Warn("checklive risk control -352, refreshing and retrying once", "room_id", roomID)
        // 刷新 query 实际使用的 signer(baseCookie 维度,codex v2 #1)
        if rs, ok := c.signerForCookie(baseCookie).(interface{ RefreshKeys() error }); ok {
            _ = rs.RefreshKeys()  // 生产 *WBISigner 满足
        }
        if c.buvids != nil { c.buvids.Invalidate(baseCookie) }  // 删对 key
        resp, err = query()  // 重新注入+签名+请求,signer 复用 baseCookie 那个
        if err != nil { return LiveInfo{}, err }
    }
    if resp.Code != 0 {
        if resp.Code == -352 {
            return LiveInfo{}, fmt.Errorf("%w: room_id=%d", ErrRiskControl352, roomID)
        }
        return LiveInfo{}, fmt.Errorf("bilibili room info error: code=%d message=%s", resp.Code, resp.Message)
    }
    // ...原解析逻辑(L114-139)不变
}
```
- **不扩大公共接口**:`biliutil.URLSigner` 保持只有 `SignURL`;用局部断言 `interface{ RefreshKeys() error }`。
- **signer 按 baseCookie 选**(v3 关键修正):`RefreshKeys` 刷的就是 `query()` 实际签名用的那个 signer,根除 codex v2 #1。附带修正:原 `signURL` 按 injected 建 signer 会在每次 buvid 刷新时建新 signer(浪费 + WBI 密钥冗余拉取),改按 baseCookie 后 signer 跨 buvid 刷新复用。
- **附带小幅重构 `signURL`**:原 `signURL(endpoint, cookieHeader)` 按 cookie 选 signer 的语义在 query 闭包内联展开(或保留 `signURL` 但内部改用 baseCookie)。两者皆可,实现时择简,但**单元测试必须断言 RefreshKeys 调用的是签名所用 signer**(codex v2 #1 测试要求)。
- **baseCookie 保留**:每次 `query()` 从 baseCookie 重新 `injectAntiRisk`(取新 buvid)+ 重新 `SignURL`(用刷新后的密钥)+ 重新请求。
- **非 -352 的 Code!=0**:维持原错误格式;**getJSON 传输错误**(网络层):直接返回,不重试。

#### 2b. `manager.go` 的 `checkOne`(v2 修正:Status 不空 + 成功重置 + 阶梯)
**方案**:
- `Manager` 加 `last352Cooldown map[string]time.Time`(冷却到期时间)+ `cooldownStep map[string]int`(连续 -352 次数)
- `checkOne` **顶部**:若 `time.Now().Before(last352Cooldown[item.ID])` → 返回 **`Status{ChannelID: item.ID, RoomID: item.LiveRoomID, Error: "risk control cooldown"}`**(v2:不空,前端可见频道身份 + 冷却原因),不发请求
- `checkOne` 捕获 `errors.Is(err, ErrRiskControl352)`:**阶梯延长**(`step=0→5m`、`1→10m`、`2+→20m`,封顶 20m),设 `last352Cooldown[id]=now+duration`、`cooldownStep[id]++`,记 INFO("cooldown applied")
- **成功 CheckLive 后**(v2 补):删除 `last352Cooldown[id]` 和 `cooldownStep[id]`(重置,避免跨成功探测累积)
- 非 -352 错误(网络抖动等)走原 WARN 路径,**不**触发冷却(避免误冷却瞬时抖动)
- 冷却 map 操作复用既有 `m.mu`
**风险**:中。手动 `Start`/`Check` 不查冷却(低频用户触发,不参与批量风控)。

### 第 3 层(节流):CheckAndStartAll 频道间加 jitter
**问题**:`maxConcurrent=8` 信号量是**并发上限**而非速率上限,≤8 频道时全瞬时齐发。
**方案**(v2 修正:**保持硬编码 8**,不读 worker.num):
- worker 闭包(manager.go:302-307)里,每个 `CheckLive` 调用前 `time.Sleep(time.Duration(rand.Intn(800)) * time.Millisecond)`(jitter 0~800ms 随机)
- **`maxConcurrent` 维持硬编码 8**(codex #6:worker.num 默认 3,且与直播检查并发概念耦合,本轮不动)
**风险**:低。
- jitter 仅作用于非录制频道的 CheckLive(录制中频道 `activeFor` 早退)。
- 最坏延迟 = 800ms × ⌈channels/8⌉,25 频道约 2.4s,远小于 30s tick。
- jitter 用 `math/rand`,非加密场景无需 seed(Go 1.20+ 自动 seed)。
