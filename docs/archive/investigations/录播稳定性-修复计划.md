## 第 1 批 · 让重启后链路可诊断、可恢复

### 🔴 异常 #6:重启后旧 pending 任务变孤儿 → session 死锁

**根因**(`internal/worker/worker.go:293-330`,`recoverRunning`):
- `recoverRunning` 只调 `store.ListRunning`(`task.go:158`,status=running),不查 pending。
- 重启后 worker 的内存 queue 清空,DB 里 pending 的 task 永远不会被 `loop()` 消费。
- 对应 session 卡 `discovered` → `ActiveLiveForChannel` 认为 active → scheduler 永久跳过该 channel。

**修复方案**:
- `recoverRunning` 处理完 running 任务后,额外恢复 pending 任务。
- **关键(codex v2 指正)**:不能复用 `ResetToPending`(`task.go:200`)——其 SQL `WHERE id=? AND status=?` 传 `StatusRunning`,对 pending task 调用会 `ErrTaskConflict`。pending task 本就是 pending 状态,只需重新塞进内存队列,不需要改 DB 状态或递增 attempt(它从未被 worker 消费)。
- 具体步骤:
  1. 新增 `store.ListPending(ctx)`(`task.go`,与 `ListRunning` 同模式,SQL `WHERE status='pending'`)。
  2. 对每个 pending task:
     - `attempt < MaxRetryAttempts` → **只 `enqueueID(task.ID)`**(不调 ResetToPending,不递增 attempt)。
     - `attempt >= MaxRetryAttempts` → `MarkFailed` **并调用 `syncSessionState`**(`worker.go:359`)同步 session 状态为 failed,否则 session 仍卡 discovered,死锁没解除。
- 文件定位:`store` 在 `internal/worker/task.go`(codex v1 指正,非 store.go)。

**TDD**(`worker_test.go`):
- pending task + attempt 未超限 → `pool.Start` 后**只入队不改 DB 状态/不递增 attempt**(断言 DB 仍是 pending,attempt 不变),最终被 fake handler 执行。
- pending task + attempt 超限 → `MarkFailed` 且 **session 状态被 syncSessionState 同步**(断言 session 变 failed,不再卡 discovered)。
- live_record 类型的 pending 孤儿也覆盖(其 handler 会因主播下播快速失败,健康路径)。

**风险**:重启涌入一批 pending 可能压垮 worker,但配合第 2 批 #5 的并发修复可接受。

**代码位置**:`internal/worker/worker.go:293`、`internal/worker/task.go`(加 ListPending)、`worker.go:359`(复用 syncSessionState)

### 🟡 异常 #8(可观测性修复):Start 错误被静默吞 → 暴露为 WARN 日志

**根因**(`internal/scheduler/scheduler.go:101-107` + `internal/live_record/manager.go:234-283` + `cmd/hikami/main.go:452-456`):
- `CheckAndStartAll` 的 `checkOne` 有**多个** `Status{Error: ...}` 返回点。
- **两处调用方都丢 Status.Error**:① scheduler cron job(`scheduler.go:107`)只打 `completed channels=N`;② main.go 启动时的 initial check(`main.go:453-455`)也只看顶层 error。
- 所以 -352、selectStream 失败等错误全部静默。

**性质说明**:本异常的修复是**可观测性改进,不是功能修复**——打 WARN 不能让"在线但不起录制"恢复,但能定位真因(配合 #7 修风控后才能真恢复)。codex v1 明确指出这点。

**修复方案**(codex v2 补全范围):
- 把错误日志下沉到 `checkOne` **内部所有 `Status{Error: ...}` 返回前**,统一 `slog.Warn` 打 channel_id + error。覆盖至少两个分支:
  1. **CheckLive 失败**(`manager.go:241`,初次探测就错,如 -352)。
  2. **Start/Check 失败**(`manager.go:271-283`,起录制失败)。
- 这样 scheduler 和 main.go initial check 两处都自动覆盖,避免重复遗漏;也不在调用方遍历(易再次漏一处)。

**TDD**(`manager_test.go`):
- mock `CheckLive` 返回 error(-352 场景)→ 调 `CheckAndStartAll` → slog 捕获 WARN(channel_id + error)。
- mock `Start` 返回 error → 同样断言 WARN。

**风险**:日志量增加。可加去重(同 channel 同 error 1 分钟内只打一次),但首版可不做。

**代码位置**:`internal/live_record/manager.go:271-283`(Start 失败分支加 slog.Warn)

### 🟡 异常 #7:live_record 的 CheckLive/GetStream 被 -352 风控

**根因**(`internal/live_record/bilibili.go:43` CheckLive + `79` GetStream + `getJSON`):
- `live_record/bilibili.go` 的请求**裸调** B 站 API:无 buvid 注入、无 WBI 签名、缺 `Referer`/`Origin`(`getJSON` 没设这些 header)。
- AGENTS.md changelog(2026-07-05)的 -352 修复只在 `channel/identify.go`,`live_record` 没下沉。
- 手动触发 + 风控敏感期批量调用暴露 500。

**修复方案**:
- `BilibiliClient`(`bilibili.go:18`)注入 `biliutil.BuvidStore` + `biliutil.URLSigner` 工厂(与 identify.go/danmaku.go 共享同一套,已下沉到 biliutil)。
- `CheckLive`(`getInfoByRoom`)和 `GetStream`(`getRoomPlayInfo`)调用前:① 注入 buvid3/buvid4;② 用 `URLSigner.SignURL` 加 `w_rid`/`wts`;③ **`getJSON` 同步 `identify.go` 的 header 策略**(浏览器 UA、`Referer: https://live.bilibili.com/<room>`、`Origin: https://live.bilibili.com`)。
- **降级策略**(与 identify 一致):buvid 获取失败或 WBI 签名失败时,`warn` 后**仍发未签名请求**(不阻断),保留容错。

**TDD**(`bilibili_test.go`,httptest mock):
- 请求 header 带了 `buvid3`/`buvid4` cookie + 浏览器 UA + Referer/Origin。
- URL query 带了 `w_rid`/`wts`。
- buvid/signer 注入失败 → 仍发请求(降级不阻断)。
- `CheckLive` 和 `GetStream` 两个端点都覆盖。

**风险**:buvid 缓存按 cookie 共享,cookie 为空时 fallback;WBI nav 拉取失败时降级。

**代码位置**:`internal/live_record/bilibili.go:18-67`(BilibiliClient 加 buvids/signers/headers)、`43`(CheckLive)、`79`(GetStream)、`getJSON`(header 同步)

## 第 3 批 · 重连可靠性

### 🔴 异常 #1:selectStream 失败 → CheckLive 误判 → 提前放弃重连

**根因**(`internal/live_record/manager.go:666-679` 重连循环 + `749-809` `decideAfterRecord`):
- 重连时 `selectStream` 失败(`bilibili stream url not found`),代码 `continue`(manager.go:671)回到循环顶部。
- 下一轮 `decideAfterRecord(ctx, task, payload, wantErr=sErr)` 被调用,它内部又调 `CheckLive`。
- 若此刻 CheckLive 返回 `live:false`(B 站 API 流断边缘态瞬时抖动),命中 `manager.go:778`(`wantErr!=nil && !live → afterRecordFinishError`)→ 直接放弃,尽管 maxReconnect 额度还剩。

**修复方案**(codex v2 补充结构性细节):
- `selectStream` 失败的路径**不走 `decideAfterRecord`**。但当前循环结构(`manager.go:614-679`)在 selectStream 失败 `continue` 后,**下一轮顶部仍会调 `decideAfterRecord`**(`:625`)用 `sErr` 再查 CheckLive——简单改 continue 不能解决。
- **必须结构性跳过下一轮顶部的判定**。三种实现选一(推荐第一种,改动最小):
  1. **引入 `skipProbeAfterSelectError bool` 标志**:selectStream 失败时置 true 并 continue;循环顶部在调 `decideAfterRecord` 前检查该标志,若 true 则跳过 decideAfterRecord、直接走 sleep + attemptsUsed++ + 重新 selectStream。
  2. 把选流失败处理成**独立 retry 子循环**(selectStream 自己重试 maxReconnect 次)。
  3. selectStream 失败时不把 `sErr` 作为下一轮 `err` 传入 `decideAfterRecord`(传 nil,但语义混乱,不推荐)。
- 只有 `attemptsUsed >= maxReconnect` 才退出。

**TDD**(`manager_test.go`):
- mock `selectStream` 连续返回 error,设 maxReconnect=3 → 断言:① 循环跑了 3 次(而非 1 次就退出);② **每次间隔 reconnectDelay**;③ **selectStream 连续失败期间 `CheckLive` 不被额外调用**(关键:验证跳过了 decideAfterRecord 的探测);④ 额度耗尽才退出。
- 边界:selectStream 失败期间主播真下播 → 重试满 3 次才放弃(可接受,延迟最多 3×reconnectDelay)。

**风险**:多试几次"无意义重连"的代价远小于"误判提前放弃漏录"(bililive-go 无上限重试)。

**代码位置**:`internal/live_record/manager.go:665-672`(selectStream 失败分支改为直接 sleep+continue)

### 🔴 异常 #2:CDN 404 重试预算/退避不足(codex v1 修正根因)

**根因修正**(codex v1 指正):原计划写"复用失效 URL"**不准确**——代码 `manager.go:666` 在**每次循环**都重新 `selectStream` 拿新流地址。真正问题是:
- `maxReconnect=3` + `reconnectDelay=10s`,3 次重试总窗口仅 ~30s。
- B 站 CDN 节点切换抖动可持续 30s-2min,3×10s 刚好卡在抖动窗口里全部失败。
- 对 `open live stream: http status 404` 这类 CDN 瞬时错误,没有特化退避。

**修复方案**:
- 保留每轮重新 `selectStream`(现状正确,不改)。
- **单一明确规则(codex v3 要求消除歧义)**:
  - **CDN transient 路径**(错误匹配 `open live stream: http status 404`、connection reset 等):**绕过 `attemptsUsed >= maxReconnect` 判断,只受独立的 `cdnRetryBudget` 控制**(初值如 5)。每次命中消耗 `cdnRetryBudget--`,delay 指数退避(base*2^n,上限 60s)。
  - **非 CDN 路径**(ffmpeg 真失败、selectStream 失败等):受 `attemptsUsed >= maxReconnect` 控制(不变)。
- 退出条件:`cdnRetryBudget <= 0`(CDN 路径)或 `attemptsUsed >= maxReconnect`(非 CDN 路径),任一耗尽即放弃。两条路径互不消耗对方的 quota。
- 伪代码(关键分支):
  ```
  segErr := m.recordAudio(...)
  if isCDNTransientError(segErr) {
      // CDN 路径:不进 decideAfterRecord,不递增 attemptsUsed
      if cdnRetryBudget <= 0 { break reconnect }
      cdnRetryBudget--
      sleep(backoff(cdnRetryBudget))  // 指数退避
      continue  // 下一轮重新 selectStream
  }
  // 非 CDN 路径:走现有 decideAfterRecord + attemptsUsed 额度
  err = segErr
  attemptsUsed++
  ```
- 不改通用 maxReconnect(避免对非 CDN 错误无限重连)。

**TDD**(`manager_test.go`):
- CDN 404 场景:mock `recordAudio` 返回 `http status 404`,设 maxReconnect=3、cdnRetryBudget=5 → 断言**实际重试 5 次**(不受 maxReconnect=3 截断),退避间隔递增。
- 非 CDN 错误:mock `recordAudio` 返回 ffmpeg 真失败 → 断言只试 maxReconnect=3 次(不受 cdnRetryBudget 影响)。
- 边界:cdnRetryBudget 耗尽 → 放弃。

**风险**:退避拉长重连窗口,主播真下播时多试更久才放弃。可接受(配合 #1 的 maxReconnect 兜底)。

**代码位置**:`internal/live_record/manager.go:614-679`(重连循环退避逻辑)

### 🔴 异常 #4:健康检测误报 `recording unhealthy`(codex v1 补全方案 A)

**根因**(`internal/live_record/manager.go:1126-1165` `checkRecordingHealth`):
- 第 1147 行硬编码检查 `audio.{container}`(`audio.m4a`)。
- 重连切分段后新 ffmpeg 写到 `audio.part.N.m4a`(`reconnectAudioSegmentPath`:819),旧 `audio.m4a` 不再增长 → 永久误报。
- `activeRecord`(`:86`)无 OutputPath 字段;`fileSizes`(map)按 channelID 存,切换文件后拿旧大值和新小值比,持续误判。

**修复方案(方案 A,codex v1+v2 补全)**:
1. `activeRecord`(`:86`)加 `CurrentOutputPath string` 字段。
2. **更新时机——两个位置都要**(codex v2 指正):
   - **首段录制前**(`manager.go:603`,`recordAudio(runCtx, stream, audioPath, ...)` 之前):设 `CurrentOutputPath = audioPath`。
   - **每个重连分段录制前**(`manager.go:674-675`,`recordAudio(runCtx, stream, segmentPath, ...)` 之前):设 `CurrentOutputPath = segmentPath`。
   - 不能只在 `addAudioSegment` 后更新(后者在 recordAudio 返回后才调,健康检测误报发生在录制中)。
3. `fileSizes`:**采用简单方案(codex v3 要求二选一写死)**——`fileSizes` 仍按 `channelID`(map[string]int64),不改 key 策略;但**每次切换 `CurrentOutputPath` 时,在同一把锁(`m.mu`)下把 `fileSizes[channelID]=0` 且 `failCount[channelID]=0`**(强制重新基线,避免"旧 56MB vs 新 15MB"错比)。`CurrentOutputPath` 只用于 `os.Stat` 取当前文件。
   - **清理策略**:`clearActive`(`manager.go:1184`)从 active map 删除时,顺带 `delete(m.fileSizes, channelID)` 和 `delete(m.failCount, channelID)`,避免 map 无限增长。
4. **Adopt 重启接管兜底**(`worker.go:336` `adoptLiveRecord` → manager.go `Adopt`):重建 activeRecord 时,CurrentOutputPath 用 glob `audio*.m4a` 取 mtime 最新的兜底(进程在跑但路径丢失的情况)。
5. `checkRecordingHealth`(`:1126-1165`)改读 `active.CurrentOutputPath`(为空时 fallback 到 glob 兜底,再 fallback 到旧的硬编码路径)。

**TDD**(`manager_test.go`):
- 模拟:录制 audio.m4a(56MB)→ 触发重连切到 audio.part.1(15MB 且增长中)→ `checkRecordingHealth` → 断言**不报 unhealthy**(检查 part.1 且按 part.1 的 size 基线,fileSize 已重置)。
- **首段场景**:新录制(无重连)CurrentOutputPath 在首段 recordAudio 前已设 → 健康检测能找到 audio.m4a。
- Adopt 兜底:activeRecord.CurrentOutputPath 为空时,glob 兜底能找到 part.1。
- 并发:更新 CurrentOutputPath(重连 goroutine)与读(健康检测 goroutine)在已有锁保护下安全。

**代码位置**:`internal/live_record/manager.go:86`(结构加字段)、`674`(recordAudio 前更新路径 + 重置 fileSize)、`1126-1165`(checkRecordingHealth 改读路径)、worker.go adopt 兜底
