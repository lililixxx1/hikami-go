## 关键架构背景(影响所有问题分析)

**hikami-go 的录制架构与 bililive-go / biliLive-tools 不同:**
- bililive-go / biliLive-tools:ffmpeg 直接 `ffmpeg -i <CDN URL>` 连 B 站 CDN,ffmpeg 自己做 HTTP 拉流。所以 `-rw_timeout` / `-reconnect` 这类 ffmpeg 输入参数有效。
- **hikami-go**:Go 用 `http.Client.Get(streamURL)` 拉流 → `io.Copy(ffmpeg.stdin, resp.Body)` → ffmpeg 从 `pipe:0` 读。**ffmpeg 与 B 站 CDN 之间没有网络 I/O**,所以 `-rw_timeout` / `-reconnect` 对 hikami 完全无效(pipe 不是网络协议)。网络超时/断流的感知全在 Go 的 `http.Client` + `io.Copy` 上。

代码位置:`internal/live_record/ffmpeg.go:24-77`(`FFmpegRecorder.Record`)。

**这意味着:**
- "socket 活着但无数据"(CDN 停发流但不断开 TCP)时,`io.Copy` 会**无限阻塞**,因为它没有读超时。当前 `FFmpegRecorder.HTTPClient` 未设置 `Timeout`(ffmpeg.go:128-131 用默认/注入的 client),`http.Client.Timeout` 又是整体超时(对长流不适用)。
- 当前依赖:B 站 CDN 主动发 `HTTP/2 INTERNAL_ERROR`(本次事件就是)或 TCP RST 让 `io.Copy` 报错返回 → 才能进入重连分支。
- 风险点:如果 B 站 CDN 静默卡死(发空数据或干脆停发但不 RST),当前架构**无法感知**,会永远挂着一个不增长的 ffmpeg。这正是计划里 `recording unhealthy: file not growing` 监控要抓的场景(代码 `manager.go:1165` 实现了"连续 3 次未增长"告警,但需要确认它对 pipe 模式有效)。

#### 根因分析(代码级,不修)

重连循环在 `internal/live_record/manager.go:614-679`:

1. **首段录制**:ffmpeg 拉 `cn-gddg-cm-01-04.bilivideo.com` 的 FLV。B 站 CDN 在 13:30:49 发 HTTP/2 `stream error: INTERNAL_ERROR`(可能是 CDN 内部切节点/限流),`io.Copy` 报错 → ffmpeg stdin 关闭 → ffmpeg 干净退出(`status=success`,**误导性**——Go 这边 io.Copy 报错被吞成了 ffmpeg 成功)。

2. **触发重连**:`decideAfterRecord`(`manager.go:749`)调 `CheckLive` 探测 → 返回 `live:true` → `wantErr != nil` → 走 `afterRecordReconnect`(`manager.go:792` 打 `stream interrupted, attempting reconnect`)。

3. **等待 reconnectDelay(10s)** → 13:31:00 调 `selectStream`(`manager.go:666` → `1018`)→ `m.client.GetStream(roomID, false, cookie)`(`bilibili.go:79`)→ B 站 `getRoomPlayInfo` 返回的 `playurl_info` 里**没有任何 stream 候选** → `bilibili.go:135` 返回 `bilibili stream url not found`。

4. **致命点**:`selectStream` 失败后,代码执行 `attemptsUsed++; continue`(`manager.go:670-671`),进入下一次循环 → 再次调 `decideAfterRecord`,此时 `wantErr = sErr`(选流错误)→ 又调 `CheckLive` → **这一刻 CheckLive 返回 `live:false`**(B 站 API 在主播实际仍在线的瞬间返回了下播状态,可能是与刚断流相关的瞬时状态/缓存)→ 命中 `manager.go:778` 的 `stream ended after record failure, finishing with error` → `afterRecordFinishError` → **整个重连循环退出,尽管 maxReconnect=3 的额度还剩 2 次**。

#### 两个复合问题

- **问题 A(直接原因)**:`getRoomPlayInfo` 在刚断流的瞬间返回空候选。B 站侧现象:流断了但直播间状态尚未稳定,playurl 接口短暂返回"无流"。
- **问题 B(放大问题)**:`selectStream` 失败时,代码 `continue` 让 `decideAfterRecord` 重新判定,而 `decideAfterRecord` 会重新 `CheckLive`。一旦 CheckLive 在断流边缘态返回 `live:false`,整个循环就提前放弃了——**重试预算(maxReconnect=3)实际上只对"选流成功后录制又断"才有效;对"选流本身就失败"的场景,一次 CheckLive=false 就全盘放弃**。

#### 与两个开源项目的对比(参考用,不修)

| 维度 | hikami-go(当前) | bililive-go | biliLive-tools |
|------|------------------|-------------|----------------|
| 选流失败后 | `continue` → 重新 CheckLive → false 就放弃 | **不存在选流步骤**—— recorder 每轮循环都重新 `GetStreamInfos()`,失败就 `time.After(5s)` 重试,**无上限**(`recorder.go:run`) | `recordRetryImmediately` 开关:以 `channelId-liveId` 去重,**单场最多重试 10 次**(`manager.ts:408-450`) |
| 重试预算 | `maxReconnect=3`,但只在"录制中断"路径生效 | 无上限,5s 间隔 | 单场 10 次 |
| 主播下播判定 | CheckLive 一次 false 就定性 | **靠 ffmpeg 持续 404 秒退**(代码注释:`ErrLiveOffline` 才停)→ 持续重试到明确下播 | 帧数停滞 + 特定错误串(下播空流)识别 |
| 失败后是否保留数据 | ✅ 保留(58MB 部分录制) | ✅(size>0 才算录) | ✅ + 后处理 `minSize` 过滤垃圾片段 |

**核心可借鉴点(待修复阶段评估):**
1. **bililive-go 模式**:把"选流"和"判定是否继续"解耦——选流失败就 sleep + 重试,**不依赖 CheckLive 来决定放弃**。只有持续重试 N 次都拿不到流(或持续 404),才认为真下播。
2. **biliLive-tools 模式**:重试以"场次"为单位限频限次(如单场最多 10 次),避免下播后无限重连空流;同时把"下播判定"从单次 CheckLive 改为"连续 M 次 CheckLive=false"或"持续 N 分钟无有效流"。
3. hikami 当前"一次 CheckLive=false 在 wantErr!=nil 时直接放弃"的语义(`manager.go:778`)对流断重连场景过于激进——这次事件就是证据:主播实际仍在线(后续调度器日志 `live:true` 持续),但因为重连窗口里的一次 API 抖动就被判了死刑。

#### 建议修复方向(本次不实施,记录待办)

- **短期**(改判定语义):`selectStream` 失败时,不要走 `decideAfterRecord` 重新 CheckLive,而是直接 `sleep(reconnectDelay); attemptsUsed++; continue`——让 maxReconnect 额度真正覆盖"选流失败"。只有 `attemptsUsed >= maxReconnect` 才放弃。
- **中期**(下播判定更稳):`decideAfterRecord` 里 `liveErr==nil && !liveInfo.Live && wantErr!=nil` 的分支,改为"连续 K 次 CheckLive=false"才算下播(参考 bililive-go 的"ffmpeg 持续 404"语义),避免单次 API 抖动误判。
- **长期**(架构):考虑给 `FFmpegRecorder.HTTPClient` 或 `io.Copy` 加基于 `ResponseHeaderTimeout` / 自定义 `net.Conn` 的读空闲超时,覆盖"CDN 静默卡死"场景(bililive-go 用 `-rw_timeout`,hikami 因 pipe 架构需在 Go 层做)。

**异常 #2 实际上是"预期内的失败"**——maxReconnect=3 都试了,每次都 404,放弃是合理的。但暴露两个问题:

#### 与两个开源项目的对比

- **bililive-go**:它不存在"选流→拉流"两步,ffmpeg 直接 `-i <URL>`,404 时 ffmpeg 秒退 → `run()` 循环 5s 后重启重新 `GetStreamInfos()` 拿新 URL。**每轮都是新 URL,不会复用失效 URL**。这是 hikami 当前可借鉴的:重连时若 404,应重新 `selectStream` 而非复用旧 stream。
- **biliLive-tools**:`recordRetryImmediately` + `channelId-liveId` 去重 + 单场 10 次 + 1 分钟以上才算有效录制。重试预算更宽松。

#### 可能原因(待查证,不修)

~~最初猜测是 worker 入队的并发竞态。~~ **17:17 的"僵尸复活"事件彻底揭示了真正根因——不是竞态,是 worker 池容量限制下的队列饥饿:**

**复活事件(17:17:04)**:弥月 session(15:17 卡在 discovered)在 17:17:04 突然 `live record started`,而**这一刻正好是弥雾重连失败、task 腾出 worker 槽位的同一秒**。南町僵尸也在同期复活(running)。鸡蛋饼(bili_3546961537927564)的任务 `task_8052453...` 至今(18:07)**仍 `pending`,已排队 2h46m**。

**根因锁定(配置 + 可见性,非代码 bug)**:
- 默认配置 `worker.live_record_num=2`(`config.go:735`),即同时最多录 2 路直播。
- 15:01-15:21 的开播潮有 **7 个主播同时上线**,但 worker 只能同时跑 2 个,其余 5 个 session 的任务进入 `pending` 队列。
- 随着前面的录制结束(失败/成功),后面的 pending 任务被依次调度:南町、弥月在 17:17 排到了(此时溪璐/三理中有一个让出槽位 + 弥雾失败腾位),鸡蛋饼仍在等。
- session 表里这些"等位"的 session 显示 `status=discovered` + `current_task_id=None`(任务 pending 时 session 未推进到 recording),**用户侧看不到"在排队"的提示**,误以为是 bug。

**修正定性**:这不是代码 bug,而是**配置默认值过小(live_record_num=2)+ 缺乏排队可见性**的组合问题:
- 修复方向 A(配置):用户测试多主播时应调高 `worker.live_record_num`(如 8-10),覆盖预期并发数。
- 修复方向 B(可见性):session 在 discovered 但 task pending 时,前端/日志应明确提示"排队中(N/M)位",而非静默显示 discovered。
- 修复方向 C(健壮性):worker 池满时应记录 WARN 日志(当前完全静默),便于排查。

#### 严重度评估(修正后)

🟢 低(就其本身而言)。这是**预期行为**(容量限制下的排队),不是数据丢失或泄漏。已"复活"的南町/弥月 session 都成功进入 recording,鸡蛋饼在 pending 排队(会陆续轮到)。但它**严重影响测试判断**——本测试前期把它误判为"僵尸泄漏 bug",浪费了排查精力。前端可见性缺失是真正的体验问题。

### 第二阶段 · 8:00 开播潮观察(重大发现 ×2)

8:00 迎来开播潮(7+ 主播同时在线),观察窗口到 8:30。期间抓到两个关键发现:

#### 🔴 异常 #5(修正异常 #3 的根因):`live_record_num` 是死配置项,实际瓶颈是 `worker.num=3`

把 `worker.live_record_num` 调到 10 **完全没生效**。代码核查(`internal/worker/worker.go`):
- 第 110 行 `for i := 0; i < workerCount; i++ { go p.loop() }` —— worker pool 启动 `workerCount`(=`worker.num`)个 goroutine,**这是唯一并发限制**。
- `grep "LiveRecordNum" internal/worker/*.go` → **零结果**。`config.Worker.LiveRecordNum` 字段存在于 config 结构体(`config.go:76`),但调度器**完全不读它**,是死配置项。

实测:8:00 开播潮 7 个主播上线,`running=3`(=worker.num 上限),`pending=4-5` 积压,session 显示 discovered。与调高 live_record_num 前的现象**完全一致**。

**修复方向**:① 短期(配置):直接调高 `worker.num`(当前=3 → 建议 8-10),这是唯一生效的旋钮。② 中期(代码):让 worker pool 真正实现 `live_record_num` —— live_record 用独立信号量限流,与通用 worker.num 解耦,避免直播录制任务挤占 normalize/asr 等其他任务的槽位。

#### 🔴 异常 #4:健康检测误报 `recording unhealthy` —— 重连切分段后监控文件路径不更新

**发生时间**:2026-07-05 20:22 起(星瞳 bili_401315430 重连后)
**现象**:星瞳 20:18 流断重连,新 ffmpeg 进程把数据写到 `audio.part.1.m4a`(持续增长,健康),但健康检测持续报 `recording unhealthy: file not growing` ×6+(RED),因为它检查的是**旧文件 `audio.m4a`**(20:18 后不再增长)。

**根因**(`internal/live_record/manager.go:1126-1165`,`checkRecordingHealth`):
- 第 1147 行:`audioPath := filepath.Join(sessionDir, "raw", "audio."+m.cfg.LiveRecord.AudioContainer)` —— **硬编码**检查 `audio.m4a`,不检查重连后的分段文件 `audio.part.N.m4a`。
- 重连时(`manager.go:674`)调用 `reconnectAudioSegmentPath(audioPath, attemptsUsed+1)`(`manager.go:819`)生成 `audio.part.N.m4a` 路径,新 ffmpeg 写到那里。
- 但 `activeRecord` 结构(`manager.go:86`)**没有 OutputPath 字段**,健康检测无法知道当前实际写到哪个分段。
- 结果:**任何发生过重连的录制都会被永久误报为 unhealthy**,直到录制结束。文件数据其实是健康的(part.N 持续增长)。

**实测数据**(20:27):
- `audio.m4a`:56MB,mtime 停在 20:18:52(ffmpeg 退出瞬间)→ 健康检测看这个,误报。
- `audio.part.1.m4a`:15.2MB,mtime 20:26:55(持续增长)→ 实际录制健康。

**修复方向**:`activeRecord` 加 `CurrentOutputPath` 字段,重连切分段时更新它;`checkRecordingHealth` 改读 `active.CurrentOutputPath` 而非硬编码 `audio.m4a`。或退一步:健康检测改为检查 session 目录下**最新的 .m4a 文件**(glob `audio*.m4a` 取 mtime 最大)。

### 第三阶段:worker.num=3→8 重启后退化(2026-07-05 20:29 起)

20:29 调 `worker.num: 3 → 8` 重启(PID 61258→78544),预期解决排队。**结果反而更糟——8 个主播在线却 0 路录制**,暴露 3 个叠加问题:

#### 🔴 异常 #6:重启后旧 pending 任务变孤儿 → session 永久卡 discovered → 死锁阻塞新录制

重启前有 3 个 pending 任务(花礼、bili_1203217682、溪璐,异常 #5 排队产生)。重启后 worker pool 的 queue channel(内存)清空,但这 3 个 task 在 DB 里仍是 pending 状态。`recoverRunning`(`worker.go:101`)只处理"ffmpeg 进程存活"的场景,**不管 pending 孤儿任务**——没有任何机制把它们重新入队。

后果链:
1. 3 个 task 永远 pending → 对应 session 永远 `discovered`
2. scheduler 的 `CheckAndStartAll` 对这 3 个 channel 调 `ActiveLiveForChannel` → 发现 discovered(算 active)→ 返回 `ErrAlreadyRecording` → **跳过,不再尝试**
3. 死锁:session 不会自己变 failed(没有 task 执行),也不会变 recording(没 worker 处理)→ 永久阻塞该 channel 的所有后续录制

**严重度**:🔴 高。这是**重启引入的状态污染**,每次重启都可能产生孤儿,且无自愈机制。修复:`recoverRunning` 应同时把 DB 里 status=pending 的 task 重新入队(或标记 failed)。

#### 🔴 异常 #7:手动触发录制 500 — `Manager.Start` 的 CheckLive 被 -352 风控

手动 `POST /api/live/:id/record/start` 调三理/岁己 → `Manager.Start` → `CheckLive`(`manager.go:355`)→ B 站 `getInfoByRoom` 返回 `code=-352` → 500。

**关键反差**:scheduler 的 `CheckAndStartAll` 也调 `CheckLive`,但**大部分成功**(日志里 `live status checked` 事件正常,live:true 判定正确)。说明 scheduler 路径和手动 handler 路径的 `CheckLive` 行为不一致——可能 scheduler 共享了 buvid/WBI 缓存(刚建立),而手动触发的高频调用触发风控。

这正是 AGENTS.md changelog(2026-07-05)记录的 -352 风控,但那次修复在 `channel/identify.go`,**`live_record/Manager` 的 `CheckLive` 没有同等的 buvid 注入 + WBI 签名加固**(`biliutil.go` 的 `GetStream` 等端点)。`live_record/bilibili.go:43` 的 `CheckLive` 直接裸调 `getInfoByRoom`。

**严重度**:🟡 中(手动触发场景)。scheduler 自动轮大部分时候 OK,但风控敏感期会批量失败。

#### 🔴 异常 #8:scheduler 判定在线却不调起录制(疑似 Start 内部静默失败)

最严重的问题:8 个主播在线(scheduler 的 `CheckAndStartAll` 的 `CheckLive` 成功判定 live:true),但 **0 路录制起来**。`live record start requested` 事件一个都没有。

可能原因(待查证):
- `CheckAndStartAll`(`manager.go:271`)调 `Start`,但 `Start` 内部 `CheckLive`(`manager.go:355`)被 -352 拒(scheduler 的 CheckLive 和 Start 内的 CheckLive 是两次独立调用,后者可能在风控窗口失败)→ 返回 error → `CheckAndStartAll` 静默吞掉(只记 Status.Error,不 slog)
- 或:`ActiveLiveForChannel`/分钟槽 UNIQUE 把它们都挡了(但 failed 状态不应被 ActiveLive 挡)

实测:`curl /api/live/bili_2030198123/status` 返回 `live:true, recording:false` —— 在线但不录制,直接证据。

**严重度**:🔴 高。这是核心功能失效——主播在播却完全不录。需深入排查 `CheckAndStartAll` → `Start` 的错误处理路径。

#### 第三阶段小结

重启后退化不是单一 bug,而是**异常 #6(孤儿)+ #7(-352)+ #8(scheduler 不起录制)**三者叠加。其中 #6 和 #8 是代码 bug,#7 是风控加固覆盖不全。第三阶段未能验证 `worker.num=8` 是否解决排队(因为根本没起录制),需先修复 #6/#8 让录制恢复,才能继续。
