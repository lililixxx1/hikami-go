## 〇、问题摘要(调研结论)

### 异常 #10(P0,核心):重连死循环占满 worker 槽 → 漏录

**现象**:流已断的频道,日志反复 `reconnect liveness probe failed, attempt anyway`,占着 worker 槽不释放,阻塞 pending 任务。

**根因**(代码取证,非猜测):
- `internal/live_record/manager.go:1025-1038` 的 `decideAfterRecord` 的 `default` 分支:`CheckLive` 探测**出错**(`liveErr != nil`,如 -352/网络抖动)+ 已有录制错误(`wantErr != nil`)时,**仍返回 `afterRecordReconnect`**(manager.go:1037),即"探测都失败了也照常重连"。
- 重连循环(manager.go:767-908)拿到 `afterRecordReconnect` 后,manager.go:873 只检查 `attemptsUsed >= maxReconnect`,**不区分**"探测明确说还在播"和"探测失败,凑合重连"。
- 后果:每次重连都对着**确实已断**的流跑 `selectStream` + `recordAudio`(manager.go:891-907),注定再失败,再触发 `decideAfterRecord`,再 `attempt anyway`……直到 `maxReconnect`(默认 3)耗尽,期间占槽 `maxReconnect × reconnectDelay`(默认 3×10s = 30s)+ 探测本身的网络耗时。
- 在 412 风控期 / CDN 不稳时,CheckLive 持续出错,该路径在每个频道上反复上演,叠加 worker.num 槽位被占 → 新开播排不进 → **漏录**。

**注意(校正调查文档措辞)**:这不是"无限循环",而是"反复占槽的低效重试"。`maxReconnect` 上限存在,但在风控期会持续触发,危害成立。修复目标:**探测错误不应等同"仍在播"**,应独立计数预算,快速耗尽释放槽位。

### 异常 #11(P1,可观测性):健康检测对 0 字节文件有盲点 → 僵尸 task

**现象**:ffmpeg 起来但从未写入数据(selectStream 拿不到有效流)时文件 0 字节,健康检测**不触发 unhealthy**。

**根因**(代码取证):
- `checkRecordingHealth`(manager.go:1405-1461)用 `info.Size() > lastSize` 判增长(manager.go:1448)。
- `updateCurrentOutputPath`(manager.go:1476-1488)切分段时把 `fileSizes[channelID]=0`、`failCount[channelID]=0` 重置(manager.go:1484-1485)。
- 首段录制开始时(manager.go:721)调 `updateCurrentOutputPath` → 基线 = 0。
- **盲点**:ffmpeg 起来但 `selectStream` 拿到的 URL 无效(返回 200 但无数据 / 连接挂起超时前),`recordAudio` 不立即失败(ffmpeg 进程挂着),文件停留在 0 字节。健康检测:0 > 0 = false → `failCount++`,3 次(= 3×60s = **3 分钟**)后才 WARN(manager.go:1453),且 **WARN 后什么都不做**(不取消 task、不标记失败),僵尸 task 继续占槽直到自然超时或重启。

### P2(低,防御性):HTTP 层风控码(412/403)纳入冷却

**现象**:异常 #9 只处理业务码 -352(`getJSON` 在 HTTP 2xx 下解析 body 的 `code` 字段),HTTP 网关层的 412/403/429 直接被 `getJSON` 折叠成 `fmt.Errorf("bilibili http status %d")`(bilibili.go:329),**不返回哨兵、不刷新密钥、不触发冷却**,频道每个 tick 都硬打。

**根因**(代码取证):
- `getJSON`(bilibili.go:311-332)对非 2xx 一律折叠成字符串错误,丢失了状态码语义。
- `CheckLive`(bilibili.go:111-157)的 -352 检测/`RefreshKeys`/`Invalidate` 重试只在 `getJSON` 返回 nil(HTTP 2xx)后发生;`getJSON` 报 HTTP 错时直接 return(bilibili.go:128-130),根本走不到 -352 分支。
- `checkOne`(manager.go:275)/`Check`(manager.go:426)用 `errors.Is(err, ErrRiskControl352)` 触发冷却,412/403 不匹配 → 落入普通 WARN(manager.go:286),**无冷却**。

## 二、异常 #10 修复:重连探测错误独立预算

### 2.1 设计

**核心思路**:把"探测明确说仍在播"和"探测失败凑合重连"区分开。后者用**独立的 `probeErrorBudget`**(默认 1),耗尽即当作"无法确认状态,保守放弃",释放槽位。

**为什么预算=1 而非 0**:CheckLive 偶发网络抖动(非风控)是真实的,允许"探测错一次 → 再试一次重连"能容忍瞬时抖动;但第二次仍探测错,基本可判定流确实已断或网络已坏,继续重连纯属浪费槽位。预算=1 是"乐观但克制"的平衡点(对齐调查文档"耗尽即放弃")。

**为什么不直接 `FinishError`**:首段已录到有效音频,放弃重连后仍应走成功收尾(`finalizeAudioSegments` + `EventLiveRecordSucceeded`),保留已录内容,而非丢弃。这与现有 `wantErr==nil` + 探测错 → `FinishSuccess`(manager.go:1029-1032)的语义一致,只是把它扩展到 `wantErr != nil` 但探测预算耗尽的情况。

> **codex 审核关注点**:此处"探测错且 wantErr != nil,预算耗尽 → FinishSuccess(收 nil err)"是否合理?理由:wantErr 是上一段 recordAudio 的错(可能就是流自然结束的 clean EOF 被包装,或 CDN 瞬时错),已录音频是有价值的;探测无法确认状态时,保留内容比丢弃更安全。**但需收 nil err 走成功路径**,与 `attemptsUsed >= maxReconnect` 时收 `carryErr` 走失败路径(manager.go:873-874)区分开。这是本计划最关键的设计决策,请重点审。

### 2.2 改动清单

#### 2.2.1 `decideAfterRecord` 签名扩展(传入选代上下文)

**问题**:`decideAfterRecord` 当前无状态,不知道这是第几次"探测失败凑合重连"。需要让它能告知调用方"这次是探测失败型重连"。

**方案**:新增返回字段不优雅(已有 decision + carryErr 两返回值)。改为:**让 `decideAfterRecord` 在 default 分支返回一个新的第四种决策 `afterRecordProbeFailReconnect`**,调用方据此递增 `probeErrorBudget`。

**`internal/live_record/manager.go:49-58`**(枚举扩展):
```go
type afterRecordDecision int

const (
	afterRecordFinishSuccess afterRecordDecision = iota
	afterRecordFinishError
	afterRecordReconnect
	afterRecordProbeFailReconnect // 异常 #10:探测失败型重连,独立预算,耗尽即收尾
)
```

**`internal/live_record/manager.go:1025-1038`**(default 分支改返回新决策):
```go

	// liveErr != nil(探测出错):
	//   wantErr == nil → 保守收尾(不丢弃已录音频);
	//   wantErr != nil → 异常 #10:不再"attempt anyway",改走独立预算的
	//     ProbeFailReconnect,耗尽则由调用方收尾(成功路径,保留已录音频)。
	if wantErr == nil {

		"channel_id", task.ChannelID, "room_id", payload.RoomID,

```

> 删除"attempt anyway"措辞,改为"retrying with probe-error budget"。

#### 2.2.2 重连循环新增 `probeErrorBudget`

**`internal/live_record/manager.go:759-766`**(预算初始化,在 `reconnect:` 循环前):
```go

// 不与 maxReconnect(明确仍开播的重连预算)混用。

```

**`internal/live_record/manager.go:863-879`**(switch decision 分支扩展):
```go
switch decision {
case afterRecordFinishSuccess:
	err = nil
	break reconnect
case afterRecordFinishError:
	err = carryErr
	break reconnect
case afterRecordReconnect:
	// 明确仍开播(Live=true)的重连:maxReconnect 预算。
	if attemptsUsed >= maxReconnect {
		err = carryErr
		break reconnect
	}
	err = carryErr
	_ = reporter.Progress(ctx, 15+attemptsUsed*5, fmt.Sprintf("reconnecting (attempt %d/%d)", attemptsUsed+1, maxReconnect))
case afterRecordProbeFailReconnect:
	// 异常 #10:探测失败型重连:probeErrorBudget 预算。
	// 耗尽则收尾,**但收尾路径取决于是否已录到有效音频**(codex v1 Critical #1):
	//   - 有有效分段(len(audioSegments)>0):走成功收尾,保留已录音频(err=nil);
	//   - 无有效分段(所有 recordAudio 都失败,0 字节):走失败路径,不送 normalize
	//     (避免空音频污染回顾;finalizeAudioSegments 对空 segments 返回 nil,
	//      会把 succeeded 状态 + enqueue normalize 推下游 —— 实质性回归)。
	if probeErrorBudget <= 0 {
		if hasRecordedAudio(audioSegments) {
			slog.Info("probe-error budget exhausted, finishing with recorded audio preserved",
				"channel_id", task.ChannelID, "room_id", payload.RoomID, "segments", len(audioSegments))
			err = nil
		} else {
			slog.Warn("probe-error budget exhausted with no valid audio, finishing with error",
				"channel_id", task.ChannelID, "room_id", payload.RoomID, "carry_error", carryErr)
			err = carryErr // 非 nil → 走 manager.go:910 失败分支,不 enqueue normalize
		}
		break reconnect
	}
	probeErrorBudget--
	err = carryErr
	_ = reporter.Progress(ctx, 15, "reconnecting after inconclusive live probe")
}
```

> **新增 helper**(`hasRecordedAudio`,manager.go 靠近 `audioFileExists`):
> ```go

> // 用于异常 #10:探测预算耗尽时决定走成功收尾(保留已录音频)还是失败路径(不送 normalize)。
> // audioSegments 已在 addAudioSegment 里用 audioFileExists(Size()>0) 过滤,这里只需判空。

>
> **关键**:`afterRecordProbeFailReconnect` 耗尽时,`err = nil`(有音频)或 `err = carryErr`(无音频)+ `break reconnect`。前者落到 manager.go:930+ 成功收尾;后者进 manager.go:910 `if err != nil` 失败分支(不 enqueue normalize)。codex v1 Critical #1 落实点。

## 三、异常 #11 修复:健康检测 0 字节盲点 + 取消僵尸 task

### 3.1 设计

**两层修复**:

1. **0 字节快速失败**:`checkRecordingHealth` 检测到文件 `Size() == 0`(首次进入健康检测的频道,基线 0;或切分段后基线重置为 0)时,**单独累计一个 `zeroSizeStreak[channelID]`**(连续 0 字节次数)。连续 N 次(用配置化的阈值,默认 2 = 2×60s = 2 分钟,比 3 分钟盲点更快)→ 触发取消。
2. **WARN 后取消**:现有 `failCount >= 3`(文件不增长)和新的 `zeroSizeStreak >= N`(0 字节)两个条件,**任一满足即取消 task**:`active.Cancel()`(manager.go:95 已存 Cancel func)。取消后 `runCtx.Done()` 传播到重连循环(manager.go:770 / 885)和 ffmpeg,task 走取消收尾路径(manager.go:911-924,成功收尾保留已录音频)。

> **codex 审核关注点**:取消走"成功收尾"而非"失败",因已录部分仍有价值(对齐 #10 决策)。但 0 字节场景**没有**已录音频,走成功收尾会生成空音频文件送下游 normalize —— 这可能污染回顾。**替代方案**:0 字节取消应走**失败**路径(MarkFailed),让 session 标记失败而非成功。本计划采纳:**0 字节 → 失败收尾**(区别于"文件曾增长后停滞" → 成功收尾保留已录)。请 codex 确认此区分是否合理。

**修正设计**(落实上面 codex 关注点):
- **0 字节取消**(`zeroSizeStreak >= N`):`Cancel()` + 标记一个哨兵错误,让 HandleTask 走**失败**路径(manager.go:910 `if err != nil` 分支),不送 normalize。
- **文件曾增长后停滞**(`failCount >= 3` 且 `fileSizes > 0`):保留现有"成功收尾"语义(有已录音频,值得保留)。

> 实现上:健康检测不能直接改 HandleTask 的 err(它在另一个 goroutine)。用 `Cancel()` + 一个新的 `activeRecord` 字段 `abortReason error`,HandleTask 的 `defer`(manager.go:610-613)或在循环退出后检查 `abortReason` 决定走成功还是失败路径。

```go
func (m *Manager) checkRecordingHealth() {
	m.mu.Lock()
	channelIDs := make([]string, 0, len(m.active))
	for id := range m.active {
		channelIDs = append(channelIDs, id)
	}
	m.mu.Unlock()

	for _, channelID := range channelIDs {
		m.checkOneChannelHealth(channelID) // 每个频道独立加锁-释放,无跨循环 defer
	}
}

// checkOneChannelHealth 处理单个频道的健康检测。所有锁操作在本函数内成对 Lock/Unlock,
// 不使用 defer(避免被误放到循环上下文)。返回是否触发了 abort(便于测试/日志聚合)。
func (m *Manager) checkOneChannelHealth(channelID string) bool {
	active, ok := m.activeFor(channelID) // 内部自管锁
	if !ok {
		return false
	}
	ctx := context.Background()
	sessionInfo, err := m.sessions.Get(ctx, active.SessionID)
	if err != nil {
		return false
	}
	sessionDir := filepath.Join(m.cfg.OutputRoot, channelID, sessionInfo.Slug)
	fallbackPath := filepath.Join(sessionDir, "raw", "audio."+m.cfg.LiveRecord.AudioContainer)
	audioPath := active.CurrentOutputPath
	if audioPath == "" {
		if globbed := globLatestAudio(m.cfg.OutputRoot, channelID, active.SessionID, m.cfg.LiveRecord.AudioContainer); globbed != "" {
			audioPath = globbed
		} else {
			audioPath = fallbackPath
		}
	}
	info, err := os.Stat(audioPath)
	if err != nil {
		m.applyHealthStat(channelID, func(stats *healthStats) {
			stats.failCount++
		})
		return false
	}

	// 计算新状态(锁外算,锁内只写)
	var abort error
	m.applyHealthStat(channelID, func(stats *healthStats) {
		if info.Size() == 0 {
			// 异常 #11:0 字节盲点。
			stats.zeroSizeStreak++
			if stats.zeroSizeStreak >= m.zeroSizeFailThreshold() && stats.abortReason == nil {
				stats.abortReason = ErrZeroByteStalled
				abort = ErrZeroByteStalled
			}
			// 0 字节不更新 fileSizes/failCount(它们基于"文件曾增长"的语义)
			stats.fileSizes = 0
			stats.failCount = 0
			return
		}
		stats.zeroSizeStreak = 0
		if info.Size() > stats.fileSizes {
			stats.failCount = 0
			stats.fileSizes = info.Size()
		} else {
			stats.failCount++
			if stats.failCount >= 3 && stats.abortReason == nil {
				stats.abortReason = ErrRecordingNotGrowing
				abort = ErrRecordingNotGrowing
			}
		}
	})

	if abort != nil && m.markAbort(channelID, abort) {
		slog.Warn("recording unhealthy, aborting task",
			"channel_id", channelID, "session_id", active.SessionID,
			"abort_reason", abort, "fail_count", m.peekFailCount(channelID))
		return true
	}
	return abort != nil
}
```

#### 3.1.5 `markAbort` helper + 阈值

**新增 helper**(manager.go,靠近 `updateCurrentOutputPath`):

```go
// markAbort 在锁已持有的前提下,为 active 记录调用 Cancel 取消 task。
// abortReason 已由 checkOneChannelHealth 在 applyHealthStat 内写入 healthStats,
// 本函数只负责触发 Cancel(幂等:Cancel 本身可重入,且 healthStats.abortReason!=nil 时不再重复设)。
// 返回 true 表示本次是首个触发者(abortReason 本次从 nil 变非 nil 的那次)。
// 调用方:**不**持有 m.mu(markAbort 内部自管锁)。
func (m *Manager) markAbort(channelID string, reason error) bool {
	m.mu.Lock()
	defer m.mu.Unlock()
	active, ok := m.active[channelID]
	if !ok || active.Cancel == nil {
		return false
	}
	active.Cancel()
	return true
}
```

## 四、P2 修复:HTTP 风控码(412/403/429)纳入冷却

### 4.1 设计

**复用 -352 的整套冷却设施**,只新增一个 HTTP 层哨兵 + 让 `getJSON` 识别状态码。

**为什么复用而非新建**:冷却的语义(频道级阶梯 5/10/20m、成功后重置、checkOne/Check 双入口跳过)与触发原因无关,只与"这个频道现在不该被探测"有关。HTTP 412 和业务 -352 都是"别再打这个频道"的信号,共用冷却 map 是 DRY 的正解。`applyCooldown352` 改名 `applyCooldownRiskControl`,map 名从 `last352Cooldown` 改 `lastRiskCooldown`(或保留旧名加注释,见 4.4)。

### 4.2 改动清单

#### 4.2.1 新哨兵

**`internal/live_record/types.go`**(在 `ErrRiskControl352` 旁):
```go

var ErrHTTPRiskControl = errors.New("bilibili http-layer risk control")
```

#### 4.2.2 `getJSON` 识别状态码

**新增 helper**(bilibili.go,靠近 `getJSON`):
```go
// httpStatusError 把非 2xx HTTP 状态码映射为错误。
// 风控相关码(412 Precondition Failed / 403 Forbidden / 429 Too Many Requests)
// 包装 ErrHTTPRiskControl 哨兵,供 checkOne/Check 识别并触发冷却;
// 其它非 2xx(5xx 服务器错误等)返回普通错误,不触发冷却(重试有值,不该惩罚)。
func (c *BilibiliClient) httpStatusError(status int) error {
	switch status {
	case http.StatusPreconditionFailed, http.StatusForbidden, http.StatusTooManyRequests:
		return fmt.Errorf("%w: status=%d", ErrHTTPRiskControl, status)
	default:
		return fmt.Errorf("bilibili http status %d", status)
	}
}
```

> 注意:`fmt.Errorf("%w: ...", ErrHTTPRiskControl, ...)` 让 `errors.Is(err, ErrHTTPRiskControl)` 成立。

#### 4.2.3 `CheckLive` 对 HTTP 风控重试一次

#### 4.2.4 `checkOne` / `Check` 扩展冷却触发

**问题**:当前 `errors.Is(err, ErrRiskControl352)` 只识别 -352。需同时识别 `ErrHTTPRiskControl`。

```go
// isRiskControlError 判断是否为风控类错误(业务码 -352 或 HTTP 层 412/403/429)。
// 两者共用同一套频道级冷却。
func isRiskControlError(err error) bool {
```


> 两处(checkOne + Check)同步改。日志措辞从"-352 risk control"改"risk control"泛化。

### 4.3 `GetStream` / `Start` / preflight 的 HTTP 风控策略(codex v1 Important #3,#4 + 决策 G 反转)

> **codex v1 决策 G 反对理由**:`getJSON` 一改,`GetStream` 实际也会返回 `ErrHTTPRiskControl` 包装错误;但 `HandleTask` / `selectStream` 不识别它,会在手动启动(`Start`)、preflight 乐观放行后的 `selectStream`、重连 selectFail 路径里继续浪费 worker 槽,且不触发冷却。**v2 采纳 codex 建议:在调用方显式识别 ErrHTTPRiskControl,触发频道冷却或快速失败**。

**调用方逐一处理**(bilibili.go 改完后,以下三处都要识别 `ErrHTTPRiskControl`):

#### 4.3.1 `Start` 路径(manager.go:531 区域,手动启动)

`Start` 内部的 CheckLive 返回 `ErrHTTPRiskControl` 时,除向上抛错外,**设置频道冷却**(避免 UI/用户反复点触发累频):
```go
// Start 内 CheckLive 后:
if err != nil {
	if isRiskControlError(err) {
		until := m.applyCooldown352(channelID) // 复用冷却,名字见 4.4
		slog.Warn("start: risk control, channel enters cooldown",
			"channel_id", channelID, "cooldown_until", until)
		return Status{}, fmt.Errorf("%w (cooldown until %s)", err, until.Format(time.RFC3339))
	}
	return Status{}, err
}
```
> 实现 `Start` 时定位其 CheckLive 调用点(本期实现阶段 grep `func (m *Manager) Start`),补此分支。

#### 4.3.2 `HandleTask` preflight(manager.go:640-647)乐观放行的修正

**现状**:preflight 探测错(含 -352/HTTP 风控)**乐观放行**(manager.go:645 "proceed optimistically"),随后 selectStream/recordAudio 失败。**问题**:HTTP 风控下,乐观放行后 selectStream 也必然拿不到流(GetStream 同样被 412 挡),整条录制失败但**不触发冷却**。

**v2 修正**:preflight 探测到 `ErrHTTPRiskControl`(或 `ErrRiskControl352`)时,**不乐观放行,直接快速失败 + 设置冷却**:
```go
preflight, preflightErr := m.client.CheckLive(runCtx, payload.RoomID, cookieHeader)
if preflightErr != nil {
	if errors.Is(runCtx.Err(), context.Canceled) {
		return runCtx.Err()
	}
	// 异常 P2:HTTP 层/业务码 风控在 preflight 命中时,不乐观放行(否则 selectStream 必败、占槽不冷却)。
	// 设置频道冷却并快速失败,让 scheduler 下次 tick 跳过该频道。
	if isRiskControlError(preflightErr) {
		until := m.applyCooldown352(task.ChannelID)
		slog.Warn("preflight live check hit risk control, skipping recording and entering cooldown",
			"channel_id", task.ChannelID, "room_id", payload.RoomID,
			"error", preflightErr, "cooldown_until", until)
		return fmt.Errorf("%w: preflight (cooldown until %s)", preflightErr, until.Format(time.RFC3339))
	}
	slog.Warn("preflight live check failed, proceed optimistically",
		"channel_id", task.ChannelID, "room_id", payload.RoomID, "error", preflightErr)
	preflight = LiveInfo{}
} else if !preflight.Live { /* 原样 */ }
```
> 关键:风控类错误**不进**乐观放行,与"网络抖动等非风控错误才乐观放行"区分。这阻止了"preflight 风控 → selectStream 必败 → 重连死循环占槽"的连锁。

#### 4.3.3 `selectStream` 在重连循环内(manager.go:800/831/891)的快速失败

**现状**:`selectStream` 返回 `ErrHTTPRiskControl` 时,被当作普通 selectFail,走 maxReconnect 预算重试 3 次(占槽 3×reconnectDelay)。

**v2 修正**:在重连循环的 selectStream 错误分支(`sErr != nil`)额外判断 `isRiskControlError(sErr)`,**立即设置冷却 + break 退出**,不耗 maxReconnect:
```go
stream, sErr := m.selectStream(runCtx, payload.RoomID, m.cookieHeaderForChannel(ctx, task.ChannelID))
if sErr != nil {
	if isRiskControlError(sErr) {
		// 异常 P2:selectStream 命中风控(GetStream 被 412 挡),不耗 maxReconnect,
		// 设置冷却后立即退出释放槽位。
		until := m.applyCooldown352(task.ChannelID)
		slog.Warn("reconnect selectStream hit risk control, aborting reconnect and entering cooldown",
			"channel_id", task.ChannelID, "room_id", payload.RoomID,
			"error", sErr, "cooldown_until", until)
		err = sErr
		break reconnect
	}
	slog.Warn("reconnect stream selection failed", "error", sErr)
	attemptsUsed++
	selectFailedPending = true
	err = sErr
	continue
}
```
> 三处 selectStream 调用点(CDN 路径 800 / selectFail 路径 831 / decideAfterRecord 路径 891)都加此判断。实现时抽 helper `handleSelectStreamRiskControl(channelID, sErr) (aborted bool)` 避免三处重复。

#### 4.3.4 `GetStream` 本身不改重试

`GetStream`(bilibili.go:187-253)**不加重试**。理由:GetStream 返回的 `ErrHTTPRiskControl` 由调用方(4.3.3 的 selectStream 判断)识别并触发冷却;GetStream 内加重试会与调用方的快速失败冲突。getJSON 改完后 GetStream 自然会返回 `ErrHTTPRiskControl` 包装错(无需 GetStream 自身改动),上层即可识别。

#### 4.3.5 `HandleTask` 首次 `selectStream` 的风控处理(codex v2 P1 必改项)

> **codex v2 P1(核心漏项)**:HandleTask 在 preflight(manager.go:640)之后、首次拉流调 `selectStream`(manager.go:655),错误时 `return err`(manager.go:657)。v2 只覆盖了 preflight(4.3.2)和重连循环三处 selectStream(4.3.3),**漏了首次 selectStream**。场景:`CheckLive` 成功(live=true,preflight 放行)但随后 `GetStream` 返回 `ErrHTTPRiskControl`(风控命中 playurl 端点)→ 直接 return err → **不进冷却**,下次 tick 该频道又被 CheckLive 放行后又 GetStream 失败。这是 v1 "preflight 后 GetStream 风控"反馈的核心路径,**必须补**。

**v3 修正**(manager.go:655-658):
```go
// 获取直播流:优先纯音频,根据配置决定回退策略
stream, err := m.selectStream(ctx, payload.RoomID, cookieHeader)
if err != nil {
	// 异常 P2(codex v2 P1):首次 selectStream 命中风控(GetStream 被 412 挡),
	// 设置频道冷却并快速失败,避免下次 tick 再次放行后又 GetStream 失败。
	if isRiskControlError(err) {
		until := m.applyCooldownRiskControl(task.ChannelID)
		slog.Warn("initial selectStream hit risk control, entering cooldown",
			"channel_id", task.ChannelID, "room_id", payload.RoomID,
			"error", err, "cooldown_until", until)
		return fmt.Errorf("%w: initial select stream (cooldown until %s)",
			err, until.Format(time.RFC3339))
	}
	return err
}
```

> 关键:`selectStream`(manager.go:1297)会包装 GetStream 错误(`audio stream not available: %w`),`errors.Is` 能透传到 `ErrHTTPRiskControl` 哨兵,`isRiskControlError` 仍成立。这堵住"CheckLive OK → GetStream 风控 → return err 不冷却"的漏洞。

**新增测试 13:`TestHandleTaskInitialSelectStreamRiskControlEntersCooldown`**(codex v2 P1 落实)

- **场景**:`statefulLiveClient` 第 1 次(preflight)返回 Live=true;`selectStream` 包装的 GetStream 用自定义 client 返回 `fmt.Errorf("%w: status=412", ErrHTTPRiskControl)`(模拟 playurl 端点风控);recorder 不应被调。

#### 4.3.6 `decideAfterRecord` 内部 `CheckLive` 的风控处理(codex v3 P1 必改项)

> **codex v3 P1(第 6 个 CheckLive 调用点漏冷却)**:`decideAfterRecord`(manager.go:987)在录制段结束/失败后探测 liveness,这是 **6 个 CheckLive 调用点里唯一还没覆盖风控冷却的**:preflight(4.3.2)、checkOne(4.2.4)、Check(4.2.4)、Start(4.3.1)、首次 selectStream(4.3.5,经 GetStream)、重连 selectStream(4.3.3,经 GetStream)都已处理,但 `decideAfterRecord` 内部的 CheckLive 探测若返回 `ErrHTTPRiskControl`/`ErrRiskControl352`,v3 只走 #10 的 probe-error budget,**不 `applyCooldownRiskControl`**。这违背 P2"HTTP 412 与 -352 共用冷却机制"的设计目标 —— 录制中的频道会在重连探测时反复触发风控却不冷却,持续占槽 + 累频。

**v4 修正**(manager.go:987,`decideAfterRecord` 内 CheckLive 调用后):
```go
liveInfo, liveErr := m.client.CheckLive(ctx, payload.RoomID, m.cookieHeaderForChannel(ctx, task.ChannelID))

// 异常 P2(codex v3 P1):decideAfterRecord 的 CheckLive 探测命中风控时,
// 设置频道冷却(与 checkOne/Check/Start/preflight 一致),避免录制中频道在重连探测时
// 反复触发风控却不冷却。注意:冷却设置后,后续决策(default 分支的 probe-error budget)
// 仍照常进行,不改变 #10 的行为 —— 冷却只影响"下次该频道进 scheduler 的跳过"。
if liveErr != nil && isRiskControlError(liveErr) {
	until := m.applyCooldownRiskControl(task.ChannelID)
	slog.Warn("decideAfterRecord live probe hit risk control, channel enters cooldown",
		"channel_id", task.ChannelID, "room_id", payload.RoomID,
		"probe_error", liveErr, "cooldown_until", until)
}
```

> **设计要点**(与 #10 解耦):冷却设置是**独立的副作用**,不影响 `decideAfterRecord` 的决策返回值。`liveErr != nil` 仍按 default 分支走 `afterRecordProbeFailReconnect`(若 `wantErr != nil`)或 FinishSuccess(若 nil)。冷却的效用体现在**下一次 scheduler tick**:该频道 `cooldownRiskUntil` 为 true → checkOne 跳过 CheckLive(manager.go:255-261),给 B 站风控一个阶梯冷却窗口(5/10/20m)。这与"录制中频道不被 scheduler 重复探测"的现有早退(activeFor@245-253)正交 —— 录制中的频道本来就在 active,不会进 checkOne;冷却是为**录制结束后**(task 退出、active 清理)的下次开播探测做准备。

  ```go
  // statusResponse 是 newStatusSequenceServer 的预设响应项。
  type statusResponse struct {
  	status int    // 0 视为 200
  	body   string
  }

  // newStatusSequenceServer 按顺序返回预设 (status, body) 响应,统计请求数。
  // 与 newSequenceServer 区别:支持设置 HTTP status code(用于测 412/403/429 风控映射)。
  // 计数用 sync.Mutex + int(不用 unsafe.Pointer)。
  func newStatusSequenceServer(t *testing.T, responses []statusResponse) (*httptest.Server, *int32) {
  	t.Helper()
  	var idx int
  	var mu sync.Mutex
  	var count int32
  	srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
  		mu.Lock()
  		defer mu.Unlock()
  		atomic.AddInt32(&count, 1)
  		if idx >= len(responses) {
  			t.Fatalf("unexpected request #%d", idx+1)
  		}
  		resp := responses[idx]
  		idx++
  		w.Header().Set("Content-Type", "application/json")
  		if resp.status != 0 && resp.status != 200 {
  			w.WriteHeader(resp.status)
  		}
  		_, _ = w.Write([]byte(resp.body))
  	}))
  	return srv, &count
  }
  ```

## 六、风险与回归点(v2 更新)

1. **`TestHandleTaskReconnectSurvivesProbeError` 行为等价性**(commit 1 最大风险):budget=1 下断言是否不变?实现时先跑现有测试,失败则在测试步骤注明断言调整理由。**v2 补充**:该测试用 `interruptingOnceRecorder`(首次失败二次成功,有有效音频)+ probeErrorClient,budget=1 下流程=首段失败→探测错(ProbeFailReconnect,budget 1→0)→二次成功(有音频)→探测错(budget=0,hasRecordedAudio=true→err=nil 收尾)。**预期断言不变**(2 输出 + err nil),实现时验证。
2. **`failCount >= 3` 从 WARN 升级为取消**(commit 2 行为变更):"文件曾增长后停滞" → 取消 + 成功收尾。可能影响"主播静音 3min"场景。**缓解**:走成功收尾(有已录音频);阈值 3×60s=3min 偏保守。codex v1 决策 D 同意此阈值。
3. **0 字节取消走失败路径**(commit 2):"ffmpeg 启动慢首包延迟"可能被误判。**缓解**:阈值 2×60s=2min 给足启动;只针对"基线=0 且持续 0",写入数据后 zeroSizeStreak 清零。codex v1 决策 E 同意阈值=2。
4. **HTTP 风控冷却与 -352 共用冷却机制**(commit 3):`applyCooldownRiskControl`(改名后)是 `step++`(叠加),412 会推高 step(20min 封顶)。期望行为(风控越严重冷却越久)。
5. **改名 352→RiskControl 触动测试断言**(commit 3):`manager_anomaly9_test.go` 所有 `cooldown352Until` 调用点同步改名。**风险**:漏改导致编译失败 → 编译期立即暴露,非静默回归。实现时全文替换 + `go build ./...` 兜底。
6. **preflight 风控快速失败 vs 乐观放行**(commit 3,codex v1 Important #4):4.3.2 把"风控类不乐观放行"与"网络抖动仍乐观放行"区分。**风险**:若 `isRiskControlError` 误判(把网络抖动当风控),会让本可录制的场次被快速失败 + 冷却。**缓解**:`isRiskControlError` 只匹配两个哨兵(精确),网络抖动不匹配。
