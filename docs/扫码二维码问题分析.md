## 1. 问题现象

B 站扫码登录功能在前端有两个独立入口,**两处都有问题**:

### 1.1 主播管理页(弹窗模式)

| 极少数情况下,canvas 区域**一片纯黑** | 间歇性,不易复现 |

## 2. 根因分析

### 2.1 调查过程(诚实记录,含误判修正)

本次调查经历三轮,前两轮的根因判断都是错的,第三轮通过诊断日志才抓到真相。

| 轮次 | 当时的判断 | 实际验证 |
|------|----------|---------|
| 第 1 轮(07-15 的 commit `a1a595d`) | "`watch(visible)` 缺 `immediate:true` → `startLogin` 永不调用" | **错**。加了 immediate 后问题仍在,只是从"永远不出现"变成"第二次才出现" |
| 第 2 轮(本次 07-18 早些时候) | "immediate watch 在 setup 同步阶段启动,canvas ref 未挂载,`nextTick` 不够" | **部分对**。改成 `onMounted` 后,但只覆盖了主播页,且实测仍有偶发全黑 |
| 第 3 轮(最终) | 通过 console 诊断日志抓到 `canvas.width=300, height=150`(浏览器默认值),且设置页用的是完全不同的组件 | **正确**(见下文) |

### 2.2 真实根因 1:HTML canvas 元素的默认尺寸陷阱(两个页面共性问题)

#### 关键证据(console 诊断日志实测)

在 `renderQRCode` 调用前打印 canvas 状态:

```
[BiliQRCode] 开始 { before: { w: 300, h: 150, display: "block" }, textLen: 131 }
```

**canvas 的 HTML `width`/`height` 属性是 300×150**,这是 HTML 规范规定的 `<canvas>` 元素**未设置 width/height 属性时的内置默认值**(与 CSS 的 width/height 无关)。

#### 机制

| 层 | 作用 |
|----|------|
| HTML `width`/`height` 属性 | 决定 canvas 的**位图坐标系**(绘图分辨率) |
| CSS `width`/`height` | 决定 canvas 的**显示尺寸**(拉伸到多大) |

原代码只在 CSS 里设了 220×220(显示尺寸),没在 HTML 属性里设位图尺寸。QRCode 库(`qrcode@1.5.4`)虽然内部会 `canvas.width = opts.width` 重设,但**在某些时序下**(组件刚挂载、样式计算抖动、库内部异步步骤之间),这个重设会发生在错误的时机,导致:

- 库在 300×150 的位图上画了为 220 设计的 QR 码 → 显示时被 CSS 强行拉伸/裁剪到 220×220 → **图像扭曲、变形、或呈现为全黑/全白**
- 库在异步绘制中途被某种重排打断 → 留下半成品 → **画面异常**

这是**间歇性**的——大部分时候库会成功重设并正常绘制,但触发条件具备时就会失败。

#### 与"第二次才正常"的关系

第一次点击时,canvas 刚挂载、Vue 渲染管线在多帧之间调度、QRCode 库的异步绘制时序敏感,任何一个环节出问题都会导致异常。第二次点击时,canvas 已经稳定存在、组件树已"热"、没有挂载抖动,所以库的绘制总能成功。

### 2.3 真实根因 2:设置页 `watch` 在 canvas 挂载前触发(设置页独有)

#### 关键代码

`AccountsCardV10.vue` 的原始 watch:

```ts
watch(
  () => props.qrSession?.url,
  async (url) => {
    if (!url || !canvasRef.value) return   // ← BUG
    await QRCode.toCanvas(canvasRef.value, url, {...})
  },
)
```

模板:

```vue
<div v-if="qrSession">              <!-- QR 区只在有 session 时才挂载 -->
  ...
  <canvas ref="canvasRef" />        <!-- canvas 在这里 -->
</div>
```

#### 时序问题

```
T0  用户点「扫码登录」
T1  emit('generate-qr') → 父组件 qrSession.value = await createQRCodeSession()
T2  qrSession.url 变化 → watch 回调同步触发
T3  回调内检查:canvasRef.value === null  ← ★ 因为 <div v-if="qrSession"> 刚被加入调度队列,canvas 还没挂载
T4  直接 return → 永远不画二维码
```

第二次点击时,canvas 已经挂载在那里(上一次虽然没画,但 `<div v-if="qrSession">` 没消失),`canvasRef.value` 已有值 → 正常画。

#### 与主播页的区别

主播页的 `BiliQRCodeLoginDialog` 用的是**对话框**(`HDialog`),组件本身是常驻的(由父组件 v-if 控制),用 `onMounted` 触发能保证 canvas 已挂载。

设置页的 `AccountsCardV10` 用的是**内联 QR 区**(`<div v-if="qrSession">`),canvas 的挂载时机由 `qrSession` 决定,`onMounted` 不能用(组件挂载时 session 还是 null,canvas 根本不在 DOM 里)。必须用 watch + 等待 DOM 更新。

### 2.4 真实根因 3:flex 容器的默认 stretch 行为(独立 UI bug)

#### 关键代码

```vue
<div style="display: flex; flex-direction: column; gap: 8px;">
  <HButton variant="secondary" size="sm" @click="emit('poll')">刷新状态</HButton>
  ...
</div>
```

#### 机制

flex 容器默认 `align-items: stretch`,**垂直布局(column)下会把子项横向拉满容器宽度**。

- 容器宽度 = 卡片内容区宽度 ≈ **1505px**(实测)
- 按钮被拉成 **1505px × 24px** 的长条
- 而按钮自身的内容("刷新状态"4 个字 + padding)只需要 ~74px

这是独立的 CSS 问题,与二维码渲染无关,但同在 QR 登录区,用户一并提出。

### 2.5 根因汇总

| # | 根因 | 影响范围 | 触发条件 |
|---|------|---------|---------|
| 1 | canvas HTML width/height 属性未设置(默认 300×150),QRCode 库时序敏感 | 主播页 + 设置页 | 组件挂载早期的时序窗口,间歇性 |
| 2 | 设置页 watch 回调在 canvas 挂载前同步触发,直接 return 漏画 | 设置页 | 首次点击必现 |
| 3 | flex 容器默认 `align-items: stretch`,按钮被横向拉满 | 设置页 | 必现 |

**两个二维码 bug(根因 1 + 2)叠加**,导致设置页现象最严重(必现);主播页只有根因 1,所以是偶发。

## 4. 验证步骤

### 4.1 自动化像素级验证方法(本次采用)

由于 a11y 快照不含 canvas、肉眼判断不可靠、手动测试不稳定(间歇性),本次用 **chrome-devtools-mcp** + `evaluate_script` 直接读 canvas 像素数据做客观判定:

```js
const canvas = document.querySelector('[role="dialog"] canvas')
const ctx = canvas.getContext('2d')
const data = ctx.getImageData(0, 0, canvas.width, canvas.height).data
let black = 0, white = 0
for (let i = 0; i < data.length; i += 4) {
  const sum = data[i] + data[i+1] + data[i+2]
  if (sum < 200) black++
  else if (sum > 600) white++
}
// 判定:black > 1000 && white > 1000 → 正常 QR 码(黑白约各半)
```

**判定标准**:
- canvas 宽高 = 200×200(设置页)或 220×220(主播页)
- `black > 1000 && white > 1000` → 正常 QR 码
- `darkRatio`(black / 总像素)≈ 0.5 → 符合 QR 码"黑白约各半"特征
- 全黑(black > 90% 总像素)或全白(white > 90%)→ 异常

## 5. 推荐修复方案

### 5.1 方案对比

| 方案 | 描述 | 优点 | 缺点 | 结论 |
|-----|------|-----|------|-----|
| **A. 显式设 canvas HTML width/height 属性 + 修两个调用点** | 模板绑 `:width` `:height`,JS 里再设一次;设置页 watch 改异步等 DOM | 根治、改动小、符合 HTML 规范 | — | ✅ **采用** |
| B. 单纯加 `nextTick` / `requestAnimationFrame` 重试 | 等更久再画 | 不改 canvas 尺寸 | 治标不治本,默认 300×150 仍是隐患 | ✗ |
| C. 用 `<img>` 替代 `<canvas>` | `QRCode.toDataURL()` 生成 data URL,绑到 `<img src>` | 完全规避 canvas 时序问题 | 改动大、失去 canvas 的样式灵活性 | ✗ |
| D. 移除父组件 v-if 让组件常驻 | 避免挂载竞态 | — | 改父组件模式,所有调用方都要改 | ✗ |

### 5.2 方案 A 实施细节

#### 5.2.1 主播页 `BiliQRCodeLoginDialog.vue`

> **落地说明（2026-07-18）**：本子节 (c) 建议把 `{ immediate: true }` 改回 `onMounted` 触发。**实际落地不采纳**：07-15 commit `a1a595d`（codex 已 APPROVED）用 `{ immediate: true }` 修复了主播页"watch 不触发"根因，运行至今无回归；改回 `onMounted` 在 `v-if` 控制的组件中时机不同，会扩大回归面却无收益。本次仅落地 (a)(b)（canvas width/height 属性 + renderQRCode 显式设位图尺寸）。详见 `plans/plan-investigations-2026-07-18.md` §3.4。

**(a) 模板:canvas 加 width/height 属性绑定**

```vue
<!-- 修复前 -->
<canvas v-show="session" ref="canvasRef" class="qr-canvas" />

<!-- 修复后 -->
<canvas v-show="session" ref="canvasRef" class="qr-canvas" :width="220" :height="220" />
```

**(b) renderQRCode:在调 QRCode.toCanvas 前显式设位图尺寸**

```ts
async function renderQRCode(text: string): Promise<void> {
  await nextTick()
  if (!canvasRef.value) {
    console.warn('[BiliQRCode] canvas 未就绪,跳过二维码渲染(组件可能已卸载)')
    return
  }
  // 显式设置 canvas 的位图尺寸(HTML width/height 属性),消除对 QRCode.toCanvas 内部时序的依赖。
  // 根因:canvas 元素挂载时 HTML width/height 属性默认 300x150(浏览器内置值),
  // QRCode.toCanvas 虽会重设,但在某些时序下(组件挂载早期/样式计算抖动)可能绘制异常,
  // 表现为 canvas 全黑。模板已绑 :width="220" :height="220",这里再显式设一次作双保险。
  canvasRef.value.width = 220
  canvasRef.value.height = 220
  await QRCode.toCanvas(canvasRef.value, text, {
    width: 220,
    margin: 1,
    errorCorrectionLevel: 'M',
  })
}
```

**(c) composable:用 `onMounted` 替代 `immediate watch`(防御性,首轮 2 改动保留)**

```ts
// useBiliQRCodeLogin.ts
watch(visible, (v) => {
  if (v) void startLogin()
  else void cleanupSession()
})   // ← 去掉 { immediate: true }

onMounted(() => {
  if (visible()) void startLogin()
})
```

#### 5.2.2 设置页 `AccountsCardV10.vue`

**(a) 抽出 `renderQRCode` 函数,等 DOM 挂载完再画**

```ts
// 修复前
watch(
  () => props.qrSession?.url,
  async (url) => {
    if (!url || !canvasRef.value) return   // ← BUG:canvas 没挂载时直接 return 永远不画
    await QRCode.toCanvas(canvasRef.value, url, { width: 200, margin: 1, errorCorrectionLevel: 'M' })
  },
)

// 修复后
async function renderQRCode(url: string): Promise<void> {
  await nextTick()
  await new Promise<void>(resolve => requestAnimationFrame(() => resolve()))
  if (!canvasRef.value) return
  canvasRef.value.width = 200
  canvasRef.value.height = 200
  await QRCode.toCanvas(canvasRef.value, url, { width: 200, margin: 1, errorCorrectionLevel: 'M' })
}

watch(
  () => props.qrSession?.url,
  async (url) => {
    if (!url) return
    await renderQRCode(url)
  },
)
```

> 为什么 `nextTick` + `requestAnimationFrame` 双等?
> - `nextTick()`:等 Vue 完成 DOM 更新 flush(`<div v-if="qrSession">` 挂载)
> - `requestAnimationFrame()`:等浏览器完成布局/绘制样式计算
> - 两者配合确保 `canvasRef.value` 已绑定、canvas 的 CSS 计算值已稳定

**(b) 模板:canvas 加 width/height 属性 + CSS 尺寸**

```vue
<!-- 修复后 -->
<canvas
  ref="canvasRef"
  :width="200" :height="200"
  style="border: 1px solid var(--border-light); border-radius: var(--radius-md); width: 200px; height: 200px;"
/>
```

## 7. 教训与反思

### 7.1 "试一次好了就以为修了"的陷阱

07-15 的修复(`a1a595d`)验证不严——开发者手动点了一次看到二维码出现就以为修了,没注意到"第二次才正常"的细节。本次用 chrome-devtools-mcp 像素级实测 + 三种场景(首次/刷新后首次/连续开关)才客观证明修复。

**对策**:时序/竞态类 bug 必须用自动化像素级验证,不能靠手动点击。

### 7.2 静默失败是排查噩梦

原代码 `if (!canvasRef.value) return` 无任何日志,用户和开发者都看不到失败。本次改 `console.warn` 让失败可见,是必要的防御性改进。

**对策**:任何"放弃操作"的 early return 都应该有可见日志,至少在开发模式下。

### 7.3 不要假设"组件名相似就是同一套代码"

调查第一轮我误以为两个入口都用 `BiliQRCodeLoginDialog`,只修了一个。直到用户反馈"设置页没修",才发现设置页用的是完全独立的 `AccountsCardV10.vue`(代码注释里明确写"不复用 useBiliQRCodeLogin:其为 dialog 驱动")。

**对策**:修 bug 前先 `grep` 所有调用点,不要凭名字猜测。

### 7.4 HTML canvas 的 width/height 属性必须显式设置

这是 HTML 规范层面的坑——`<canvas>` 不设 width/height 属性时默认 300×150(位图坐标系),与 CSS 的 width/height(显示尺寸)是两回事。任何用 canvas 绘图的代码都应该在模板里直接绑定这两个属性,不要依赖绘图库内部重设。

**对策**:团队规范——`<canvas>` 元素必须显式设置 HTML width/height 属性。
